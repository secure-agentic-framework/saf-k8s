# SAF-K8S-0604-002 - ML-BOM (ML Bill of Materials) generation

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.4 - SBOMs and Vulnerability Intelligence
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations deploying ML models must generate ML Bills of Materials that extend standard SBOMs to capture model lineage, training data provenance, hyperparameter configurations, framework versions, training infrastructure identity, and dataset composition. ML-BOMs must conform to CTA-2114 or an equivalent standard and must be attached to model artifacts throughout their lifecycle.

## Objective

Enable end-to-end ML supply chain transparency by documenting not only software dependencies but also the data, configuration, and infrastructure inputs that produced each model artifact.

## Rationale

Standard SBOMs capture software dependencies but not the ML-specific inputs that determine model behavior. Without ML-BOMs, organizations cannot trace a model back to its training data, identify which models are affected by a poisoned dataset, verify that training was performed on authorized infrastructure, or demonstrate compliance with data governance requirements. This creates an invisible supply chain for the most consequential component of AI workloads.

## Scope

Applies to all ML models trained or fine-tuned by the organization and deployed to Kubernetes clusters. Includes models served via inference endpoints, embedded models in application containers, and models used in batch processing pipelines. Pre-trained third-party models must have ML-BOMs obtained from the provider or reconstructed from available documentation.

## Enforcement

- Primary Enforcement Point: ML training pipeline output stage and model registry admission policy.
- Implementation Surface: ML-BOM generation tooling conforming to CTA-2114, CycloneDX ML-BOM extensions for encoding model lineage metadata, training pipeline completion step integration for ML-BOM generation, model registry storage and attachment of ML-BOMs to model artifacts, ML-BOM fields capturing model lineage (parent model, training run ID), training data provenance (dataset identifiers, versions, hashes), hyperparameter configurations, framework versions (PyTorch, CUDA, cuDNN), training infrastructure identity (cluster ID, node pool, GPU type), and dataset composition (class distributions, data source breakdown).
- Owner Primary: ML Platform Engineering
- Owner Supporting: DevSecOps / Release Engineering; Data Governance
