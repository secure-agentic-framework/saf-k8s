# SAF-K8S-0304-010 - Pod resource requests and limits specification

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.4 - Secure Defaults and Resource Constraints
- Maturity: Foundational
- Classification: baseline

## Statement

All pods must define explicit CPU, memory, and ephemeral-storage resource requests and limits for each regular and init container so the scheduler can place workloads accurately and the kubelet can enforce per-container resource boundaries.

## Objective

Prevent individual workloads from consuming unbounded node resources by requiring explicit per-container resource declarations.

## Rationale

Without explicit resource requests and limits, a single compromised or misconfigured workload can starve co-located pods, trigger evictions, or destabilize nodes through CPU, memory, or disk exhaustion. Pod-level resource specifications are the first enforcement point for bounding each workload's impact.

## Scope

Applies to all regular workload pods and init containers in production clusters. GPU resource requests must be included where applicable, but GPU quota governance is handled separately.

## Enforcement

- Primary Enforcement Point: Policy engine rules, CI/CD manifest validation, and admission checks that require per-container resource requests and limits.
- Implementation Surface: Pod spec resources.requests and resources.limits fields for CPU, memory, ephemeral-storage, and GPU extended resources where applicable.
- Owner Primary: Platform Engineering
- Owner Supporting: Application Development teams; AI/ML Platform team (for GPU resource requests)
