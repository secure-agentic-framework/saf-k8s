# SAF-K8S-0704-019 - Per-workload cloud identity binding for managed Kubernetes workloads

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.4 - Cloud Provider Security Integration
- Maturity: Foundational
- Classification: baseline

## Statement

Organizations must bind cloud API permissions to individual Kubernetes workloads through approved per-workload identity mechanisms rather than sharing node-scoped credentials across pods.

## Objective

Ensure workloads that need cloud API access receive credentials through auditable per-workload identity bindings so cloud permissions are scoped to the intended workload and service account.

## Rationale

Per-workload cloud identity mechanisms such as IRSA, EKS Pod Identity, Workload Identity Federation, and Entra Workload ID reduce lateral movement by attaching cloud permissions to specific workloads instead of every pod on a node. This makes cloud access independently attributable to the workload identity and complements metadata endpoint protections.

## Scope

Applies to pods and service accounts that require cloud API access on managed Kubernetes platforms. Covers workload identity federation setup, service-account-to-role bindings, and platform-specific per-workload identity mechanisms.

## Enforcement

- Primary Enforcement Point: Per-Workload Cloud Identity Binding
- Implementation Surface: OIDC federation setup, workload identity bindings, service account annotations, pod identity admission or webhook configuration, and workload inventory review.
- Owner Primary: Cloud Platform Engineering
- Owner Supporting: Platform Engineering; Security Engineering
