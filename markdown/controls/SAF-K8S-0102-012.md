# SAF-K8S-0102-012 - etcd health monitoring and alerting

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must implement continuous etcd health monitoring with alerts for quorum loss, storage quota pressure, replication lag, and disk or peer latency degradation so control-plane availability threats are detected before they cause write outages.

## Objective

Detect etcd degradation early enough to preserve control-plane availability and prevent operational faults from becoming security-impacting outages.

## Rationale

An unhealthy etcd cluster directly undermines Kubernetes control-plane availability. Without explicit monitoring and alerting, quorum loss, latency spikes, or storage pressure can progress into write outages that block security-critical actions such as certificate rotation, policy updates, and incident response.

## Scope

Applies to all production clusters running self-managed etcd and to managed services where provider-exposed etcd or control-plane health telemetry is available. Covers health endpoints, quorum status, latency, and storage pressure alerting. Excludes compaction and defragmentation maintenance operations.

## Enforcement

- Primary Enforcement Point: Audit / Telemetry
- Implementation Surface: Monitoring and alerting infrastructure for etcd health endpoints, quorum metrics, storage quota metrics, disk latency, and peer latency.
- Owner Primary: Platform Engineering
- Owner Supporting: Site Reliability Engineering; Security Operations
