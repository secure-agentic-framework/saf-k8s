# SAF-K8S-0903-001 - Adversarial example defenses at the serving layer

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.3 - Inference Resilience, Adversarial Defense, and Resource Controls
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Inference serving infrastructure must deploy adversarial input defenses including input transformation-based detection, perturbation detection, and adversarial input rejection policies, and must integrate explainable AI (xAI) capabilities for security monitoring while restricting xAI outputs to prevent model reverse engineering.

## Objective

Detect and reject adversarial inputs at the serving layer before they reach the model, and leverage model explanations to identify anomalous predictions while protecting explanation outputs from exploitation.

## Rationale

Adversarial examples are carefully crafted inputs designed to cause model misclassification or targeted misbehavior. Without serving-layer defenses, these attacks succeed at the application layer even against robust models. Explainable AI can help detect anomalous predictions from adversarial or poisoned inputs, but xAI outputs themselves reveal model internals that attackers can exploit for model extraction or evasion refinement.

## Scope

Applies to all production inference endpoints on Kubernetes that serve models vulnerable to adversarial manipulation. Covers input preprocessing defenses, adversarial detection middleware, xAI integration for anomaly detection, and xAI output access controls. Excludes adversarial training during model development, which is covered as an infrastructure requirement only.

## Enforcement

- Primary Enforcement Point: Inference preprocessing pipeline configuration, adversarial detection middleware or sidecar deployment, and xAI endpoint access control policies.
- Implementation Surface: Inference preprocessing pipeline with input transformation defenses (JPEG compression, spatial smoothing, input quantization), adversarial perturbation detector model deployment as sidecar or middleware, adversarial input rejection and logging configuration, explainable AI (xAI) endpoint deployment (SHAP, attention maps, feature attributions) with separate authorization and aggressive rate limiting, forensic sample retention with PII-aware redaction, and adversarial robustness evaluation suites (ART, Foolbox).
- Owner Primary: Security Engineering
- Owner Supporting: ML Platform Engineering; ML Engineering
