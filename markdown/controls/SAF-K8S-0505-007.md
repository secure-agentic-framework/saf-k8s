# SAF-K8S-0505-007 - Administrative API server access path via bastion or VPN

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.5 - API Server and Service Exposure Protection
- Maturity: Practitioner
- Classification: baseline

## Statement

Administrative access to Kubernetes API server endpoints must traverse bastion hosts, VPN connections, or equivalent managed private access paths rather than direct public internet exposure.

## Objective

Ensure human and automation administrators reach the API server only through controlled private access paths.

## Rationale

Even when API server endpoints are restricted, administrative workflows can still reintroduce exposure if operators, CI/CD systems, or support personnel connect over unmanaged public paths. Bastions, VPNs, and equivalent managed private access paths create a chokepoint for authentication, logging, and operational control.

## Scope

Applies to all production Kubernetes clusters. Managed Kubernetes services must use private endpoint or authorized network features. Self-managed clusters must place API servers on private networks with bastion or VPN access. Non-production clusters with access to production secrets or infrastructure must also restrict API server access.

## Enforcement

- Primary Enforcement Point: Bastion host access controls, VPN access controls, and equivalent managed private administrative access pathways.
- Implementation Surface: Bastion host deployments with MFA-protected SSH access, VPN configurations for kubectl and automation access, private access gateways, SSH tunneling patterns, and access logs demonstrating that administrative API usage traverses managed access points.
- Owner Primary: Platform Engineering
- Owner Supporting: Network Engineering; Security Engineering; Cloud Platform Engineering
