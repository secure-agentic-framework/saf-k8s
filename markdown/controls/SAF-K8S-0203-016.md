# SAF-K8S-0203-016 - Node time synchronization with authoritative time sources

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.3 - Host OS and Kernel Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes nodes must synchronize system time using organization-approved authoritative time sources, must monitor for clock drift beyond organization-defined thresholds, and must alert or remediate when synchronization fails.

## Objective

Preserve trustworthy time on Kubernetes nodes so security telemetry, distributed tracing, certificate validation, token lifetime enforcement, and forensic reconstruction remain accurate across the cluster.

## Rationale

Accurate time is a foundational dependency for multiple Kubernetes and AI platform security controls. Clock skew breaks cross-node audit log correlation, distorts OpenTelemetry span ordering and latency analysis across ML pipelines, causes TLS certificate NotBefore and NotAfter validation failures, and can cause projected service account tokens or OIDC tokens to be rejected incorrectly or accepted beyond intended expiry windows. During incident response, timestamp drift undermines forensic sequence reconstruction and weakens the evidentiary value of collected node, container, and network artifacts.

## Scope

Applies to all production Kubernetes worker and control-plane nodes, including GPU nodes, training nodes, and inference-serving nodes. Nodes in managed Kubernetes services satisfy this control through provider-managed time synchronization only when provider documentation or attestation confirms the service and the organization monitors observed drift or synchronization health using available telemetry. Development clusters may use different time-source hierarchies only with documented exceptions.

## Enforcement

- Primary Enforcement Point: Node operating system time synchronization service configuration and monitoring integrated into node provisioning, compliance checks, and alerting.
- Implementation Surface: Chrony, systemd-timesyncd, ntpd, or equivalent node time synchronization services; organization-approved NTP source configuration; drift and synchronization health monitoring; node compliance scans; and alerting or automated remediation workflows for unsynchronized nodes.
- Owner Primary: Platform Engineering
- Owner Supporting: OS/Infrastructure; Security Engineering; Security Operations (SOC)
