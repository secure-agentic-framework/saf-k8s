# SAF-K8S-0905-001 - AI system lifecycle classification

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.5 - AI Supply Chain and Model Lifecycle
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must classify each AI system as either train-verify-lock (model fixed at deployment) or continuous learning (model updates post-deployment) and document the classification in the AI system registry using standardized labels before production deployment on Kubernetes.

## Objective

Ensure each AI system has a documented lifecycle classification that drives appropriate security control selection, with classification labels applied before production deployment to enable risk-calibrated enforcement.

## Rationale

Train-verify-lock systems and continuous learning systems have fundamentally different risk profiles. Accurate classification is the prerequisite for applying the correct security controls. Without documented classification, there is no basis for choosing between integrity-focused controls for static models versus update governance controls for continuously learning models.

## Scope

Applies to all AI systems deployed on Kubernetes before production deployment. Covers the classification decision and registry documentation. Excludes pre-deployment model development activities and the enforcement of the control profile itself (see SAF-K8S-0905-015).

## Enforcement

- Primary Enforcement Point: Model deployment admission control that validates system classification label and corresponding control profile assignment before allowing production deployment.
- Implementation Surface: AI system registry with classification labels (train-verify-lock, continuous-learning), Kubernetes pod labels (ai.kais.io/update-model), model deployment admission control validating classification labels, file integrity monitoring for train-verify-lock models, model update authorization workflows for continuous-learning systems, drift monitoring dashboards, and periodic classification review processes.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Security Engineering; ML Engineering; Governance

- Framework: EU AI Act 2024/1689
- Requirement Title: Accuracy, robustness and cybersecurity
- Relation Type: direct
- Strength: strong
- Applicability: required
- Strength Reason Code: exact-text-match

#### Framework Mapping Notes

High-risk AI systems shall be resilient against attempts by unauthorised third parties to alter their use, outputs or performance by exploiting system vulnerabilities. The technical solutions aiming to ensure the cybersecurity of high-risk AI systems shall be appropriate to the relevant circumstances and the risks. The technical solutions to address AI specific vulnerabilities shall include, where appropriate, measures to prevent, detect, respond to, resolve and control for attacks trying to manipulate the training data set (data poisoning), or pre-trained components used in training (model poisoning), inputs designed to cause the AI model to make a mistake (adversarial examples or model evasion), confidentiality attacks or model flaws.

Risk-appropriate cybersecurity is the fundamental SAF-K8S design principle. Controls are tiered by maturity level and archetype to match circumstances and risk profiles.
