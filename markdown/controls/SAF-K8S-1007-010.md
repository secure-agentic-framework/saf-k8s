# SAF-K8S-1007-010 - Pre-upgrade Kubernetes API compatibility testing for AI workloads

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.7 - Cluster Lifecycle and Asset Inventory
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must perform compatibility testing against the target Kubernetes version for AI operators, custom resources, training workflows, and inference services before production upgrades, and must block production upgrades until test results confirm removed or changed APIs will not break required workload behavior.

## Objective

Validate AI workloads and platform components against upcoming Kubernetes versions before production upgrades are approved.

## Rationale

Even when deprecated API usage is identified, production failures still occur if teams do not verify migrated manifests, operators, admission webhooks, and workload flows against the target Kubernetes version. Pre-upgrade compatibility testing catches API removal breakage before upgrades impact production AI services.

## Scope

Applies to all Kubernetes clusters scheduled for version upgrades. Testing must validate AI-specific operators, CRDs, training jobs, inference serving paths, GPU scheduling behavior, policy enforcement, and admission workflows against the target version.

## Enforcement

- Primary Enforcement Point: Pre-upgrade compatibility test gates and upgrade approval checklists.
- Implementation Surface: Target-version test clusters such as kind or minikube, CI-based compatibility suites that deploy AI operators and workloads against the target Kubernetes version, integration tests for training job creation, inference service deployment, GPU allocation, policy enforcement, and admission webhook validation, and upgrade readiness gates tied to test outcomes.
- Owner Primary: Platform Engineering
- Owner Supporting: AI/ML Engineering; DevOps; Security Engineering
