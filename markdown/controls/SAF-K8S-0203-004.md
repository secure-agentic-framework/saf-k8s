# SAF-K8S-0203-004 - Secure boot and verified boot chain enforcement

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.3 - Host OS and Kernel Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes nodes must enable UEFI Secure Boot or equivalent verified boot chain mechanisms to ensure only signed bootloaders and kernels execute, establishing hardware-rooted trust in the node boot process.

## Objective

Establish hardware-rooted trust in the node boot process so that only verified, signed software executes from firmware through kernel initialization.

## Rationale

Without verified boot, an attacker with host access can replace the kernel or bootloader with a compromised version that disables security controls before the container runtime starts. Secure Boot validates cryptographic signatures at each stage of the boot chain, preventing tampered kernels from executing and ensuring that all subsequent security controls (container isolation, runtime monitoring, mandatory access control) operate on a trusted foundation.

## Scope

Applies to all production cluster nodes where the hardware and OS support UEFI Secure Boot. Cloud environments must enable Secure Boot on instance types that support it. Nodes on hardware that does not support Secure Boot must document this as a risk acceptance with compensating controls such as measured boot with remote attestation.

## Enforcement

- Primary Enforcement Point: Cloud instance launch templates with Secure Boot enabled and node image build pipeline with signed kernel verification.
- Implementation Surface: UEFI Secure Boot configuration, cloud instance launch templates (AWS, GCP, Azure Secure Boot settings), node image build pipeline boot chain verification, and measured boot attestation where available.
- Owner Primary: Cloud Infrastructure
- Owner Supporting: OS/Infrastructure; Security Engineering
