# SAF-K8S-0203-002 - Kernel parameter hardening via sysctl

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.3 - Host OS and Kernel Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes nodes must apply hardened kernel parameters via sysctl configuration covering network stack protections (IP forwarding restrictions, reverse path filtering, SYN flood mitigation), memory protections (ASLR enforcement, restricted core dumps), and namespace isolation settings appropriate for containerized workloads.

## Objective

Harden the Linux kernel attack surface on cluster nodes by enforcing security-relevant kernel parameters that limit network-level attacks, memory exploitation, and inter-namespace information leakage.

## Rationale

Default kernel parameters are tuned for general-purpose compatibility rather than security. Without hardened sysctl settings, nodes are vulnerable to SYN flood denial-of-service, IP spoofing through disabled reverse path filtering, memory exploitation through disabled ASLR, and cross-namespace information leakage. Container workloads that share the host kernel inherit these weaknesses, making kernel-level hardening a critical defense layer beneath container isolation.

## Scope

Applies to all cluster node kernels in production environments. Specific sysctl values must be validated against workload requirements, as some AI/ML networking configurations (e.g., RDMA, high-performance networking) may require adjusted network stack parameters with documented risk acceptance. Managed services where kernel parameters are provider-controlled require provider hardening attestation.

## Enforcement

- Primary Enforcement Point: Node provisioning templates applying sysctl configuration via /etc/sysctl.d/ drop-in files or cloud-init scripts.
- Implementation Surface: Sysctl configuration files (/etc/sysctl.d/), kernel parameter hardening values, cloud-init or configuration management provisioning, CIS Benchmark verification, and RDMA/high-performance networking parameter tuning for AI workloads.
- Owner Primary: Platform Engineering
- Owner Supporting: OS/Infrastructure; Security Engineering
