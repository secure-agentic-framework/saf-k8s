# SAF-K8S-0104-009 - Feature gate lifecycle transition tracking across Kubernetes upgrades

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.4 - CIS Benchmarks and Patch Management
- Maturity: Practitioner
- Classification: baseline

## Statement

Production Kubernetes clusters must track feature gate lifecycle transitions across Kubernetes upgrades and must remove, disable, or otherwise resolve deprecated, removed, or graduated feature gate settings so explicit gate configuration does not mask version drift or unsupported behavior after upgrade.

## Objective

Prevent stale or incompatible feature gate settings from persisting across Kubernetes version changes and creating opaque, unsupported, or insecure cluster behavior.

## Rationale

Feature gates change lifecycle state over time: they graduate, rename, deprecate, or disappear. If operators do not track those transitions during upgrades, clusters can retain obsolete gate settings that hide configuration drift, break support assumptions, or unintentionally preserve non-standard behavior after a feature reaches general availability or is removed.

## Scope

Applies to production cluster upgrade planning and validation for all control-plane and node component feature gate settings, including managed services where providers change default gate states across versions.

## Enforcement

- Primary Enforcement Point: Upgrade planning and post-upgrade configuration review for feature gate lifecycle changes.
- Implementation Surface: Upgrade runbooks, feature gate lifecycle tracking records, version-to-version gate transition reviews, and validation steps confirming deprecated or graduated gate settings were removed or updated during upgrade execution.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Architecture Review
