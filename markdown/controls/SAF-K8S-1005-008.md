# SAF-K8S-1005-008 - AI-specific incident response playbooks for Kubernetes

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.5 - Incident Response for Kubernetes
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must develop and maintain AI-specific incident response playbooks for Kubernetes environments covering compromised training jobs, model poisoning detection, GPU cryptomining, and inference endpoint abuse.

## Objective

Ensure that AI workload security incidents have purpose-built response playbooks with specialized detection and containment procedures tailored to AI-specific incident types.

## Rationale

AI workload incident types differ from standard Kubernetes security events: compromised training jobs require pipeline suspension, model poisoning requires hash verification and rollback, GPU cryptomining requires driver-level detection, and inference endpoint abuse requires rate limiting and model version quarantine. Without purpose-built AI incident playbooks, responders apply generic procedures that fail to address these specialized detection and containment requirements.

## Scope

Applies to all Kubernetes clusters hosting AI workloads in production. AI-specific playbooks must address the four named incident types at minimum. Playbooks must be tested through tabletop or live exercises at least annually.

## Enforcement

- Primary Enforcement Point: IR playbook repository, SOC runbook management system, and exercise/drill scheduling process.
- Implementation Surface: AI-specific IR playbook documentation (compromised training job, model poisoning detection, GPU cryptomining, inference endpoint abuse), GPU telemetry anomaly triggers from DCGM exporter, model accuracy drift alert triggers.
- Owner Primary: Security Operations
- Owner Supporting: AI/ML Engineering; Platform Engineering; Legal
