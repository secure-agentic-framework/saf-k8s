# SAF-K8S-0201-012 - Kubelet anonymous access and read-only port lockdown

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.1 - Kubelet and Node Configuration Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes nodes must disable anonymous kubelet API access and disable the kubelet read-only port (10255) so unauthenticated access paths do not remain available on any node.

## Objective

Eliminate unauthenticated kubelet access paths that expose workloads, logs, and node state to network-reachable attackers.

## Rationale

Anonymous kubelet access and the legacy read-only port bypass normal access-control checks. Locking down these paths is a distinct endpoint exposure obligation from webhook authn/authz enforcement.

## Scope

Applies to all kubelet instances on production cluster nodes, including dedicated GPU and AI inference nodes. For managed Kubernetes services where kubelet flags are provider-controlled, compensating controls such as node network isolation and provider-documented hardening attestation must be in place. Non-production clusters that process sensitive data or connect to production networks are also in scope.

## Enforcement

- Primary Enforcement Point: Node provisioning templates, kubelet configuration files, and configuration management tooling (e.g., Ansible, Terraform, cloud-init).
- Implementation Surface: KubeletConfiguration API settings (webhook authentication, webhook authorization, anonymous access, readOnlyPort), node provisioning templates, CIS Benchmark verification tooling (kube-bench), and kubelet configuration management automation.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Cloud Infrastructure
