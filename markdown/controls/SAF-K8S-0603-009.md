# SAF-K8S-0603-009 - In-toto and SLSA provenance attestation generation

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.3 - Attestation, Provenance, and Cryptographic Assurance
- Maturity: Advanced
- Classification: baseline

## Statement

All container images promoted to production must have in-toto attestations generated during the CI/CD build process that record SLSA provenance metadata, build-time metadata, and builder identity, and those attestations must be attached to the image digest as signed OCI artifacts.

## Objective

Provide tamper-evident provenance for each promoted image so reviewers can verify how the image was built, from which source inputs, and by which approved builder before the artifact is trusted downstream.

## Rationale

Image signatures identify a signer, but they do not prove which source, build invocation, or builder produced the image. Provenance attestations close that gap by binding the promoted image digest to the build facts needed to detect tampering in the software supply chain.

## Scope

Applies to all container images in production and staging registries, including model-serving images, training images, and infrastructure operator images. Third-party vendor images must have equivalent provenance attestations verified or compensating controls documented.

## Enforcement

- Primary Enforcement Point: CI/CD pipeline provenance generation and registry attachment step.
- Implementation Surface: in-toto attestation generation, SLSA provenance predicate creation, cosign attest or equivalent signing workflows, and OCI artifact attachment of provenance attestations to promoted image digests.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: Security Engineering; Platform Engineering
