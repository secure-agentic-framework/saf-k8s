# SAF-K8S-0101-005 - Streaming connection idle timeout enforcement

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.1 - Kubernetes API Server Security
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must configure the API server streaming connection idle timeout (--streaming-connection-idle-timeout) to automatically terminate idle streaming connections, including exec, attach, and port-forward sessions, within an organization-defined maximum idle period.

## Objective

Limit the window during which abandoned or forgotten interactive sessions remain open to the cluster, reducing the risk of session hijacking, credential reuse, and unmonitored privileged access through stale exec or port-forward tunnels.

## Rationale

Streaming connections (kubectl exec, attach, port-forward) create long-lived authenticated channels into pods. An idle session left open by an administrator or CI pipeline can be hijacked if the client workstation or runner is compromised. The default timeout of 4 hours is excessive for most operational patterns. A shorter idle timeout forces re-authentication and reduces the exposure window. This is especially important for AI clusters where exec sessions into training pods or notebook servers may carry access to model weights, training data, or GPU resources.

## Scope

Applies to all production Kubernetes clusters. Covers the API server --streaming-connection-idle-timeout flag. For managed Kubernetes services, verify whether the provider exposes this setting or applies a default timeout, and document as a provider-managed control if not configurable.

## Enforcement

- Primary Enforcement Point: API Server Configuration
- Implementation Surface: API server manifest or managed-service configuration (--streaming-connection-idle-timeout flag), audit log monitoring for long-lived streaming connections.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering
