# SAF-K8S-0101-003 - Encryption at rest for Secrets and sensitive API resources

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.1 - Kubernetes API Server Security
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must encrypt Secrets and other sensitive API resources at rest using EncryptionConfiguration with a supported encryption provider (KMS v2 preferred for new deployments), and must verify that targeted resources are stored as ciphertext in etcd.

## Objective

Protect sensitive Kubernetes API resources from exposure through direct etcd access or etcd backup compromise by ensuring resource data is encrypted at rest using organization-approved cryptographic providers.

## Rationale

Kubernetes Secrets are base64-encoded but not encrypted by default in etcd. An attacker or insider with access to etcd data, whether directly or through backups, can read all Secrets in plaintext. EncryptionConfiguration targets specific API resources for encryption before they are written to etcd. KMS v2 is preferred over KMS v1 (deprecated since Kubernetes 1.28, disabled by default since 1.29) because it uses a single DEK per API server, reducing KMS call volume and improving performance. This control addresses API-resource-level encryption; see SAF-K8S-0102-001 for etcd storage-layer and provider envelope encryption, and SAF-K8S-0403-001 for KMS key lifecycle management requirements.

## Scope

Applies to all production clusters. Covers Kubernetes Secrets at minimum; organizations should evaluate extending EncryptionConfiguration to ConfigMaps, custom resources, and other API resources that contain sensitive data, or use the wildcard resource (`*.`) to encrypt all resources. For managed Kubernetes services, applies to enabling and verifying provider-managed API-resource encryption settings.

## Enforcement

- Primary Enforcement Point: API Server Configuration
- Implementation Surface: EncryptionConfiguration resource applied to API server, encryption provider selection (KMS v2 preferred), and periodic verification that targeted resources are encrypted.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Key Management/Cryptography
