# SAF-K8S-0303-001 - Seccomp profile enforcement

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.3 - Mandatory Access Controls
- Maturity: Practitioner
- Classification: baseline

## Statement

All containers must run with a seccomp profile of at least RuntimeDefault, and workloads with elevated risk or specialized syscall requirements must use custom seccomp profiles that restrict allowed syscalls to the minimum set needed for operation, with profiles audited and version-controlled.

## Objective

Limit the kernel attack surface available to each container by enforcing seccomp system call filtering, preventing exploitation of unused or dangerous syscalls.

## Rationale

The Linux kernel exposes over 300 syscalls, many of which are unnecessary for containerized workloads and have historically been targets for privilege escalation exploits (e.g., keyctl, ptrace, mount). Without seccomp enforcement, a compromised container can invoke any syscall the kernel supports, dramatically expanding the attack surface. The RuntimeDefault profile blocks the most dangerous syscalls, while custom profiles provide workload-specific hardening.

## Scope

Applies to all containers in production clusters. The RuntimeDefault profile is the minimum baseline. Custom profiles are required for workloads that need syscalls blocked by RuntimeDefault or for high- security workloads that should be restricted beyond RuntimeDefault. Unconfined seccomp is prohibited without documented exception.

## Enforcement

- Primary Enforcement Point: Pod securityContext.seccompProfile field (type: RuntimeDefault or Localhost), enforced by PSA (Restricted level) and policy engine rules.
- Implementation Surface: Pod securityContext.seccompProfile field (RuntimeDefault, Localhost), Security Profiles Operator (SPO) for custom SeccompProfile CRD deployment and management, seccomp profile JSON files in version control, Pod Security Admission (Restricted level enforces seccomp), policy engine rules denying Unconfined seccomp, container runtimes (containerd, CRI-O) for seccomp enforcement, GKE Sandbox (gVisor) as alternative isolation on GKE Autopilot, and SPO installation on EKS/AKS self-managed node groups.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
