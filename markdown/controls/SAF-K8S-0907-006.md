# SAF-K8S-0907-006 - Training dataset access restriction across storage backends

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.7 - Feature Store Security and Data Access Controls
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Access to training datasets stored in persistent volumes, object storage, and feature stores must be restricted through Kubernetes-native and external authorization controls so that only approved workloads and principals can read or mount training data for their authorized team, project, or namespace scope.

## Objective

Enforce least-privilege access to training datasets across storage backends used for AI training and preprocessing.

## Rationale

Training datasets contain sensitive data and proprietary information. Without fine-grained access controls on persistent volumes, object stores, and feature stores, any workload or principal with broad cluster access can read or exfiltrate training data across teams or tenants.

## Scope

Applies to training dataset storage used by AI workloads on Kubernetes, including persistent volumes, object storage, and feature stores. Covers Kubernetes RBAC, storage-level access policies, external authorization, and workload identity-based access to data stores. Excludes model registry pull access and namespace deployment authorization for model serving.

## Enforcement

- Primary Enforcement Point: Kubernetes RBAC, storage access policies, workload identity authorization, and feature store access policies governing training data access.
- Implementation Surface: Kubernetes RBAC restricting PVC mounts to authorized training namespace service accounts, cloud IAM policies via workload identity federation for object storage access, feature store RBAC for dataset-backed feature views, and external authorization controls for training data access requests.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Data Engineering; Security Engineering; Platform Engineering
