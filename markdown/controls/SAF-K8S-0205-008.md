# SAF-K8S-0205-008 - kube-proxy or service proxy mode selection governance

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.5 - kube-proxy and Node Networking Security
- Maturity: Practitioner
- Classification: baseline

## Statement

Clusters must select and document the kube-proxy mode or kube-proxy replacement implementation based on a security and performance tradeoff analysis appropriate to the cluster workload and networking architecture.

## Objective

Ensure the service proxy layer is deliberately chosen based on documented security and performance tradeoffs rather than inherited defaults.

## Rationale

iptables, IPVS, and eBPF-based replacements have different kernel surface, auditability, and operational implications. Deliberate mode selection with documented tradeoffs is independently auditable from hardening because it governs architectural choice before implementation-specific safeguards are applied.

## Scope

Applies to all production clusters using kube-proxy or kube-proxy replacement implementations. Clusters using eBPF-based CNI plugins that fully replace kube-proxy (e.g., Cilium kube-proxy replacement) must document the replacement and apply equivalent hardening. Managed Kubernetes services must document the provider-selected mode and any available hardening options.

## Enforcement

- Primary Enforcement Point: Cluster provisioning templates, architecture decision records, and service proxy mode configuration at cluster bootstrap.
- Implementation Surface: kube-proxy mode configuration (iptables or IPVS) or kube-proxy replacement selection (for example Cilium replacement), plus architecture decision records and provider-mode verification.
- Owner Primary: Platform Engineering
- Owner Supporting: Network Engineering; Security Engineering
