# SAF-K8S-0405-009 - API impersonation RBAC restriction

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.5 - Identity Abuse Detection and Mitigation
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must restrict the Kubernetes API impersonation capability (the impersonate verb on users, groups, serviceaccounts, userextras, and uid resources) to explicitly authorized subjects only, using narrowly scoped RBAC roles and resourceNames that limit which identities may be impersonated.

## Objective

Prevent unauthorized identity assumption through the Kubernetes impersonation API by tightly constraining which subjects can impersonate which identities.

## Rationale

The impersonate verb allows a caller to act as another identity without possessing that identity's credentials. If broadly granted, it bypasses normal RBAC boundaries and enables direct privilege escalation into cluster-admin or sensitive service accounts. Restricting the grant itself is a separate control claim from detecting how approved impersonation is used.

## Scope

Applies to all clusters. Covers RBAC grants for impersonating users, groups, serviceaccounts, userextras, and uid resources. Legitimate CI/CD, multi-tenant controller, and operator use cases must be explicitly approved and scoped to named targets.

## Enforcement

- Primary Enforcement Point: RBAC / Authorization
- Implementation Surface: ClusterRoles, Roles, RoleBindings, and ClusterRoleBindings granting impersonate with scoped resourceNames and approved impersonator registry.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; Security Operations
