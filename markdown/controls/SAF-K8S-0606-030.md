# SAF-K8S-0606-030 - GitOps configuration revision and deployment history traceability

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.6 - CI/CD and GitOps Pipeline Security
- Maturity: Practitioner
- Classification: baseline

## Statement

GitOps deployment processes must preserve configuration revision history and deployment traceability records so deployed state can be reconstructed and correlated to specific Git revisions, rendered manifests, and deployment events during audits, incident response, and rollback analysis.

## Objective

Ensure GitOps-managed deployments retain configuration revision and deployment history records sufficient to reconstruct deployed state and trace configuration changes to specific revisions and rollout events.

## Rationale

Even when release packages are identifiable, teams cannot reconstruct deployed state or investigate a faulty rollout without revision history linking rendered configuration, Git revisions, and deployment events. This is a separate traceability obligation from package/version retention.

## Scope

Applies to production and staging GitOps deployments and to revision history, rendered manifests, deployment events, and traceability records needed to reconstruct deployed configuration state.

## Enforcement

- Primary Enforcement Point: GitOps deployment history and retained configuration revision records.
- Implementation Surface: Git revision metadata, rendered-manifest history, deployment controller history, environment revision records, and change traces linking reconciled state to specific commits and rollout events.
- Owner Primary: Platform Engineering
- Owner Supporting: DevSecOps / Release Engineering; Security Engineering
