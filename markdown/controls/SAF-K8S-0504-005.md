# SAF-K8S-0504-005 - Service mesh tuning for AI workloads

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.4 - Zero Trust Architecture and Service Mesh
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Service mesh deployments must account for AI workload requirements including evaluating and mitigating mTLS overhead on high-throughput GPU training traffic, managing sidecar proxy resource consumption to avoid contention with GPU workloads, and hardening ingress and API gateway configurations for model serving endpoints against inference-specific attacks.

## Objective

Ensure service mesh security controls are effectively applied to AI workloads without degrading training performance or GPU utilization, while hardening model serving endpoints.

## Rationale

AI workloads have unique performance characteristics that conflict with standard service mesh configurations. mTLS encryption overhead on high-throughput GPU training traffic (multi-gigabit NCCL all-reduce operations) can significantly degrade training performance. Sidecar proxies consume CPU and memory resources on GPU nodes where those resources are critical for data loading and preprocessing. Model serving endpoints face unique threats including prompt injection, model extraction via repeated inference queries, and adversarial input attacks that require specialized ingress hardening.

## Scope

Applies to Kubernetes clusters running AI/ML workloads within a service mesh. Covers GPU training workloads with high-bandwidth inter-node communication, inference serving endpoints exposed via ingress, and AI pipeline orchestration services. Performance exemptions from mTLS must be documented with compensating controls.

## Enforcement

- Primary Enforcement Point: Service mesh sidecar resource limits, mesh policy exceptions for high-throughput training traffic, and API gateway/ingress configuration for model serving endpoints.
- Implementation Surface: Service mesh sidecar injection annotations (sidecar.istio.io/inject: false for training pods), Cilium eBPF service mesh (sidecar-free), Envoy sidecar resource limit tuning (CPU, memory, concurrency, buffer settings), CNI-level WireGuard encryption as compensating control for sidecar-exempt training pods, NCCL traffic exemption from proxy paths, API gateway configurations for inference endpoints (request size limits, response size limits, per-client rate limiting, input schema validation), and performance benchmarks comparing AI training throughput with and without mesh mTLS.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Network Engineering; Security Engineering; Platform Engineering
