# SAF-K8S-0501-001 - Default deny ingress and egress network policies

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.1 - Network Policies
- Maturity: Foundational
- Classification: baseline

## Statement

Every namespace in production Kubernetes clusters must have a default deny NetworkPolicy for both ingress and egress traffic, with explicitly defined allow rules for each approved communication path.

## Objective

Establish a zero-trust network baseline at the namespace level so that only explicitly authorized traffic flows are permitted between workloads.

## Rationale

Without default deny policies, any pod can communicate with any other pod in the cluster, enabling lateral movement after an initial compromise. Attackers who gain access to a single workload can pivot to databases, control-plane components, or exfiltration endpoints without network-layer restriction. Default deny policies force all traffic to be explicitly authorized, dramatically reducing the blast radius of a compromise.

## Scope

Applies to all namespaces in production Kubernetes clusters and to non-production clusters that host sensitive data or AI workloads. System namespaces (kube-system, monitoring) must also have default deny policies with explicit allow rules for required control-plane and observability traffic.

## Enforcement

- Primary Enforcement Point: Kubernetes NetworkPolicy objects applied at namespace creation via admission controllers or GitOps policy enforcement.
- Implementation Surface: Kubernetes NetworkPolicy objects (default deny ingress/egress with empty podSelector), admission controllers (Kyverno generate rules, OPA Gatekeeper mutation webhooks) for automatic policy creation at namespace creation, CNI plugins enforcing NetworkPolicy (Cilium, Calico, AWS VPC CNI network policy agent), GitOps policy enforcement pipelines, connectivity testing tools (cilium connectivity test, netshoot), and namespace audit scripts verifying default-deny coverage.
- Owner Primary: Network Engineering
- Owner Supporting: Platform Engineering; Security Engineering
