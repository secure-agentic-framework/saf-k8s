# SAF-K8S-0702-025 - Tenant admission boundary enforcement for cross-tenant resource isolation

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.2 - Namespace Isolation and Multi-Tenancy
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must use admission controls to reject cross-tenant resource references and other tenant-isolation violations before workloads or resources are created or updated in the cluster.

## Objective

Prevent cross-tenant resource-manipulation paths by enforcing tenant isolation rules at admission time.

## Rationale

RBAC alone cannot stop every isolation-model violation because workloads and resource definitions can still request unsafe configurations or cross-tenant references. Admission control blocks those noncompliant resources before scheduling and makes tenant-boundary policy enforceable at creation time.

## Scope

Applies to all multi-tenant Kubernetes clusters. Admission policies must reject cross-tenant resource references and other workload or resource definitions that violate the tenant isolation model. Shared-service exceptions must be explicitly documented and reviewed.

## Enforcement

- Primary Enforcement Point: Admission control policies enforced at create and update time.
- Implementation Surface: Admission control policies such as Kyverno or OPA/Gatekeeper preventing cross-namespace resource references, Pod Security Admission enforcement, image allowlisting, required label validation, and admission test cases confirming rejection of violating resources.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Network Engineering
