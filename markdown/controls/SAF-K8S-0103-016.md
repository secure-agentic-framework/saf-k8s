# SAF-K8S-0103-016 - Controller-manager and scheduler loopback bind-address enforcement

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.3 - Controller-Manager, Scheduler, and Cloud Controller Security
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must bind the controller-manager and scheduler secure serving endpoints to the loopback address (127.0.0.1 or ::1) unless an explicitly approved alternate architecture requires a different local-only exposure pattern.

## Objective

Prevent network-based access to controller-manager and scheduler secure endpoints by constraining those components to local-only bind addresses.

## Rationale

Even when secure ports use authenticated serving, exposing controller-manager and scheduler endpoints on network-accessible interfaces increases attack surface for reconnaissance, abuse, and misconfiguration. Enforcing loopback bind addresses keeps these control-plane endpoints reachable only to local processes or explicitly brokered access paths.

## Scope

Applies to all self-managed Kubernetes clusters. Covers `--bind-address` and equivalent local-interface exposure settings on kube-controller-manager and kube-scheduler. For managed Kubernetes services, this is provider-managed and should be documented in the shared responsibility model.

## Enforcement

- Primary Enforcement Point: Controller-manager and scheduler bind-address configuration
- Implementation Surface: Controller-manager and scheduler static pod manifests or systemd unit configurations, secure-port bind-address settings, and local reachability validation.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering
