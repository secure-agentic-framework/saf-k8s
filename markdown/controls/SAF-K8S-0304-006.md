# SAF-K8S-0304-006 - Host volume mount restriction

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.4 - Secure Defaults and Resource Constraints
- Maturity: Foundational
- Classification: baseline

## Statement

Host volume mounts (hostPath) must be denied by default through admission control, with exemptions only for documented infrastructure workloads (log collectors, monitoring agents) that have compensating controls and are tracked in the exemption register.

## Objective

Prevent containers from accessing the host node's filesystem through hostPath volume mounts, blocking a direct path to node compromise and cross-workload data access.

## Rationale

HostPath volumes mount arbitrary host directories into containers, enabling read/write access to node filesystems including /etc/shadow, kubelet credentials, and container runtime sockets. An attacker with write access to a hostPath mount can modify host binaries, cron jobs, or systemd units to achieve node-level persistence. HostPath also exposes data from other containers on the same node through shared filesystem paths.

## Scope

Applies to all pods in production clusters. Infrastructure workloads that legitimately require host filesystem access (log collectors, node monitoring agents) must be isolated in dedicated namespaces with formal exemptions and compensating controls documented in the exemption register (SAF-K8S-0301-004).

## Enforcement

- Primary Enforcement Point: Policy engine rules (OPA Gatekeeper, Kyverno) denying hostPath volume mounts, with scoped exemptions for infrastructure workloads in dedicated namespaces.
- Implementation Surface: Policy engine rules (OPA Gatekeeper, Kyverno) denying hostPath volume mounts with scoped exemptions for infrastructure workloads, exemption register for hostPath exceptions (log collectors, monitoring agents), and audit logging for any pod admitted with hostPath volumes.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering
