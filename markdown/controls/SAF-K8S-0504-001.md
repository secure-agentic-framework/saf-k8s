# SAF-K8S-0504-001 - Zero trust networking principles for Kubernetes

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.4 - Zero Trust Architecture and Service Mesh
- Maturity: Practitioner
- Classification: baseline

## Statement

Kubernetes clusters must implement zero trust networking principles including identity-centric access control where workload identity rather than network location determines access, continuous verification of communication authorization, microsegmentation of all workload communication, and least-privilege network access granting only the minimum required connectivity.

## Objective

Shift network security from perimeter-based trust to identity-based, continuously verified access control for all intra-cluster communication.

## Rationale

Traditional perimeter-based network security assumes that traffic within the cluster is trusted, which is invalid in multi-tenant, multi-workload environments where any compromised pod becomes an insider threat. Without identity-centric access, any workload that can reach a service endpoint is implicitly authorized. Without continuous verification, a compromised token or identity remains usable until expiry. Without microsegmentation and least-privilege networking, lateral movement is unconstrained once an attacker gains any foothold.

## Scope

Applies to all production Kubernetes clusters. Zero trust principles must be implemented progressively, starting with workload identity and microsegmentation. Full implementation includes continuous authorization verification via service mesh or equivalent. Non-production clusters with access to production data must also adopt zero trust networking.

## Enforcement

- Primary Enforcement Point: Service mesh authorization policies, workload identity (SPIFFE/mTLS), network policies, and continuous policy evaluation at the data plane.
- Implementation Surface: Service mesh authorization policies (Istio AuthorizationPolicy, Linkerd ServerAuthorization, Cilium L7 CiliumNetworkPolicy), workload identity infrastructure (SPIFFE/SPIRE, service mesh mTLS certificates), Kubernetes NetworkPolicy for microsegmentation, Istio PeerAuthentication (STRICT mTLS mode), continuous policy evaluation at the data plane (Envoy proxy, eBPF datapath), and zero trust posture assessment scripts checking mTLS coverage and AuthorizationPolicy presence per namespace.
- Owner Primary: Security Architecture
- Owner Supporting: Network Engineering; Platform Engineering
