# SAF-K8S-0904-003 - Notebook and experimentation environment security

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.4 - AI Pipeline Orchestration and Experimentation Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Interactive notebook environments must enforce per-user isolation, kernel security hardening, resource limits, network restrictions, approved base images, extension vetting, and data access controls that prevent unauthorized dataset access, model exfiltration, and credential harvesting from interactive environments.

## Objective

Contain the security risks inherent in interactive notebook environments by isolating users, restricting capabilities, and controlling data access from experimentation workspaces.

## Rationale

Notebook environments provide interactive code execution with direct access to cluster resources, training data, and model artifacts. Without user isolation, one researcher can access another's data and credentials. Without network restrictions, notebooks can exfiltrate data to external endpoints. Without extension vetting and approved base images, malicious code can execute in the kernel. Notebooks are a common vector for credential harvesting and model theft.

## Scope

Applies to all interactive notebook deployments on Kubernetes used for ML experimentation (e.g., JupyterHub, VS Code Server, Zeppelin, RStudio Server). Covers user pod isolation, kernel configuration, resource limits, network policies, image provenance, extension management, and data access from notebook pods. Excludes production pipeline execution environments.

## Enforcement

- Primary Enforcement Point: Notebook platform spawner configuration, notebook pod security context, Kubernetes NetworkPolicy resources for notebook namespaces, and image allow-list admission policies.
- Implementation Surface: Notebook platform spawner configuration (JupyterHub, VS Code Server, Zeppelin, RStudio Server), per-user notebook pod isolation with dedicated service accounts, pod security context and resource limits (CPU, memory, GPU), Kubernetes NetworkPolicy restricting notebook pod egress to authorized data sources and package repositories, approved base image allowlists enforced by admission control, notebook extension vetting procedures, terminal access controls, egress data transfer monitoring, and data access audit logging from notebook environments.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Security Engineering; Data Engineering
