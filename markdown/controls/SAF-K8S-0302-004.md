# SAF-K8S-0302-004 - Host namespace isolation enforcement

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.2 - Security Contexts and Capabilities
- Maturity: Foundational
- Classification: baseline

## Statement

Workloads must not use hostPID, hostIPC, or hostNetwork unless a documented exception with compensating controls is approved, and clusters must enforce this restriction through admission control that denies host namespace access by default.

## Objective

Prevent containers from sharing the host's PID, IPC, and network namespaces, maintaining the isolation boundary between container workloads and the underlying node.

## Rationale

Host PID namespace access allows a container to see and signal all host processes, enabling process injection and credential harvesting. Host IPC namespace access permits shared memory manipulation of host processes. Host network namespace access bypasses network policies and exposes the container to the node's network interfaces, including the kubelet API and metadata services. These capabilities collectively enable container escape, lateral movement, and cloud credential theft.

## Scope

Applies to all workloads in production clusters. Infrastructure components that legitimately require host namespace access (e.g., CNI plugins, node monitoring agents) must be isolated in dedicated namespaces with formal exemptions. Host namespace restrictions are enforced at both Baseline and Restricted PSS levels.

## Enforcement

- Primary Enforcement Point: Pod Security Admission (Baseline and Restricted levels), policy engine rules denying hostPID, hostIPC, and hostNetwork except for explicitly exempted workloads.
- Implementation Surface: Pod spec fields (hostPID, hostIPC, hostNetwork), Pod Security Admission (Baseline and Restricted levels), policy engine rules denying host namespace access with scoped exemptions, Kubernetes audit log queries for host namespace admission events, exemption register for infrastructure components (CNI plugins, node monitoring agents, NVIDIA MPS daemon), network policies for compensating controls on exempted pods, and AppArmor/seccomp profiles limiting exempted pods to required operations.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
