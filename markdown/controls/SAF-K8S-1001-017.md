# SAF-K8S-1001-017 - Audit capture of admission, authorization, and privileged API decisions for AI workloads

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.1 - Logging and Audit
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Production Kubernetes clusters running AI workloads must configure kube-apiserver audit policies to capture admission decisions, authorization-sensitive requests, privileged API operations, and metrics API access affecting AI workload access, isolation, and change authorization at audit levels sufficient for investigation.

## Objective

Ensure that the audit trail preserves the control-plane decisions and privileged API activity that determine whether AI workloads, identities, and tenants were authorized, admitted, or modified in accordance with policy.

## Rationale

AI incidents often hinge on who approved or modified access, whether an admission controller allowed a risky object, and which privileged APIs were queried or changed. These events require richer audit capture than routine request metadata because they establish the decision path behind access, isolation, and high-impact configuration changes.

## Scope

Applies to production Kubernetes clusters hosting AI workloads and to non-production AI environments where security investigation or compliance evidence is required. Managed Kubernetes services remain in scope where customers control workload-level audit coverage decisions.

## Enforcement

- Primary Enforcement Point: API server audit policy rules and audit levels for admission, authorization, metrics, and privileged control-plane actions.
- Implementation Surface: Kube-apiserver audit policy YAML covering RBAC, admission webhooks, SubjectAccessReview flows, privileged configuration changes, and metrics API access.
- Owner Primary: Security Operations
- Owner Supporting: Platform Engineering; AI/ML Engineering
