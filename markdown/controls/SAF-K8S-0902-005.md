# SAF-K8S-0902-005 - Multi-cluster inference routing and failover security

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.2 - Inference Server and Model Serving Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Multi-cluster inference deployments must implement secure request routing with authenticated cross-cluster communication, model version consistency verification across clusters, failover controls that prevent routing to clusters with stale or unverified models, and unified access control that enforces consistent authorization policies regardless of which cluster serves a request.

## Objective

Ensure that multi-cluster inference routing maintains model integrity, authentication, and authorization consistency across all serving clusters, preventing request routing to clusters with stale, unverified, or compromised models.

## Rationale

Multi-cluster inference is deployed for geographic latency reduction, regulatory compliance, high availability, and capacity scaling. Without model version consistency verification, failover can route traffic to a cluster serving a stale model that lacks security patches or guardrail updates. Without authenticated routing, the routing layer becomes a target for traffic redirection attacks. Without consistent authorization policies, access controls diverge across clusters, creating authorization bypass opportunities.

## Scope

Applies to all multi-cluster inference deployments including geographic routing, active-active, active-passive failover, and canary cross-cluster patterns. Covers global load balancers, multi-cluster ingress controllers, service mesh gateways, and any component that routes inference requests across Kubernetes cluster boundaries.

## Enforcement

- Primary Enforcement Point: Global load balancer or multi-cluster ingress routing configuration, model version verification at each cluster, GitOps-delivered authorization policies across all serving clusters, and failover health checks incorporating model version validation.
- Implementation Surface: Global load balancers, multi-cluster ingress controllers (Istio multi-cluster, Envoy), service mesh gateways, model version registry with OCI digest tracking per cluster, GitOps multi-cluster delivery (Argo CD ApplicationSets, Flux multi-cluster), federated identity for cross-cluster routing authentication (SPIFFE/SPIRE), OpenTelemetry distributed tracing across cluster boundaries, per-cluster model staleness monitoring, and failover health checks incorporating model version validation and security posture assessment.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Network Engineering; Security Engineering; Security Operations
