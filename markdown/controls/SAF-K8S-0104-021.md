# SAF-K8S-0104-021 - Kubernetes component version skew compliance

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.4 - CIS Benchmarks and Patch Management
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must maintain component versions within the Kubernetes version skew policy across the API server, kubelet, controller-manager, scheduler, and administrative clients.

## Objective

Prevent interoperability failures and control-plane instability caused by unsupported version skew between Kubernetes components and clients.

## Rationale

Even when clusters are nominally supported, unsupported skew between API servers, kubelets, control-plane components, and administrative clients can create API incompatibilities, unstable behavior, and failed operations. Enforcing skew compliance keeps component interactions inside tested compatibility boundaries.

## Scope

Applies to all production clusters. Covers Kubernetes control-plane components, worker node kubelets, and administrative client versions that must remain within published skew policy boundaries.

## Enforcement

- Primary Enforcement Point: Component compatibility governance
- Implementation Surface: Version skew compliance reports, component inventory comparisons against skew policy, and periodic checks of administrative client compatibility.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Change Management
