# SAF-K8S-0301-001 - Pod Security Standards level assignment

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.1 - Pod Security Standards and Admission
- Maturity: Foundational
- Classification: baseline

## Statement

Organizations must classify all namespaces against the three Pod Security Standards levels (Privileged, Baseline, Restricted) and must assign each production namespace to the most restrictive level that its workloads support, defaulting to Restricted unless a documented exception justifies a less restrictive level.

## Objective

Ensure every namespace operates under an explicitly chosen Pod Security Standards level so that workloads are constrained to the minimum set of host and kernel privileges they require.

## Rationale

Without explicit Pod Security Standards assignment, namespaces default to an unrestricted posture that permits privileged containers, host namespace access, and other dangerous configurations. Attackers who compromise a workload in an unrestricted namespace can escalate to node-level access, pivot across the cluster, and exfiltrate data from co-located tenants.

## Scope

Applies to all namespaces in production clusters and to non-production clusters that share infrastructure with production or contain sensitive data. System namespaces (kube-system) may use the Privileged level with a documented exception.

## Enforcement

- Primary Enforcement Point: Pod Security Admission controller via namespace labels, supplemented by policy engine validation (OPA Gatekeeper, Kyverno) for custom constraints.
- Implementation Surface: Pod Security Standards (PSS) namespace labels (pod-security.kubernetes.io/enforce, audit, warn), Pod Security Admission (PSA) controller, namespace classification tooling (kubectl label --dry-run=server), policy engines (OPA Gatekeeper, Kyverno) for custom constraints beyond built-in PSA, GitOps namespace manifests, and cluster-wide namespace compliance scanning scripts.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
