# SAF-K8S-0102-019 - etcd endpoint network isolation from worker and workload traffic

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must isolate etcd at the network level so that only authorized control-plane components can reach etcd endpoints and worker nodes, workloads, and external networks cannot directly connect to etcd ports.

## Objective

Prevent RBAC bypass through direct etcd reachability by enforcing network isolation around etcd endpoints.

## Rationale

Direct network reachability to etcd bypasses API server authentication and authorization, allowing a compromised node or workload to read, modify, or delete cluster state directly. Restricting etcd ports to authorized control-plane paths preserves the API server as the sole managed access path to cluster state.

## Scope

Applies to all production Kubernetes clusters. Covers host firewalls, security groups, routing boundaries, and equivalent node-level controls that restrict etcd port access. For managed Kubernetes services, applies to verifying that provider-managed etcd is not exposed to customer workloads or networks.

## Enforcement

- Primary Enforcement Point: Node and control-plane network boundary enforcement
- Implementation Surface: Node firewall rules, cloud security groups, control-plane network placement, and periodic reachability assessment from worker and workload paths.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Network Engineering
