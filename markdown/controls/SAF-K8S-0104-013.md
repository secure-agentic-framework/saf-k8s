# SAF-K8S-0104-013 - Kubernetes upgrade strategy, validation, and rollback planning

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.4 - CIS Benchmarks and Patch Management
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must maintain a documented upgrade strategy that includes deprecation tracking, pre-upgrade impact analysis, compatibility validation, upgrade runbooks, rollback procedures, and post-upgrade validation steps for control-plane components, nodes, and critical add-ons.

## Objective

Ensure Kubernetes upgrades are planned and executed with controlled risk rather than ad hoc change activity that causes outages or breaks dependent workloads.

## Rationale

Remaining on supported versions requires more than version inventory: organizations need repeatable upgrade planning, API deprecation tracking, validation of workload and add-on compatibility, and tested rollback paths. Without that discipline, upgrades are delayed or create avoidable service disruption.

## Scope

Applies to all production clusters and to routine as well as urgent version upgrades that affect Kubernetes components, node pools, and critical platform add-ons.

## Enforcement

- Primary Enforcement Point: Change Management / Operations
- Implementation Surface: Upgrade planning documentation, validation workflows, rollback procedures, and post-upgrade verification integrated into change execution.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Change Management
