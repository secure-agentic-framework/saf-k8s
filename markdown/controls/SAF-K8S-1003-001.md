# SAF-K8S-1003-001 - STRIDE threat modeling for Kubernetes AI systems

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.3 - Threat Modeling Methodologies
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must conduct STRIDE-based threat modeling for Kubernetes clusters hosting AI workloads, systematically analyzing each Kubernetes component, including the API server, etcd, kubelet, scheduler, container runtime, and AI-specific operators, for Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, and Elevation of Privilege threats, and must document identified threats with corresponding mitigations.

## Objective

Systematically identify and document threats to Kubernetes AI infrastructure using the STRIDE methodology to ensure comprehensive threat coverage and traceable risk mitigation.

## Rationale

Kubernetes AI environments introduce complex trust boundaries across control plane components, GPU schedulers, model serving endpoints, and training orchestrators. Without systematic STRIDE analysis mapped to each component, organizations overlook threat categories such as spoofing of AI operator identities or tampering with training job specifications that ad-hoc threat assessments fail to surface.

## Scope

Applies to all Kubernetes clusters hosting AI workloads. Threat modeling must cover core Kubernetes components, AI-specific custom resources and operators, GPU device plugins, and model serving infrastructure. Must be updated when significant architectural changes occur.

## Enforcement

- Primary Enforcement Point: Security architecture review process, threat model documentation repository, and architecture change review gates.
- Implementation Surface: STRIDE threat model documentation repository, security architecture review process and change review gates, threat-to-mitigation mapping documents covering Kubernetes components (API server, etcd, kubelet, scheduler, container runtime) and AI-specific operators (Kubeflow, KServe, GPU device plugin, model serving controllers) analyzed for Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, and Elevation of Privilege threat categories, data flow diagrams with trust boundary annotations, and periodic review cadence aligned with architecture changes.
- Owner Primary: Security Architecture
- Owner Supporting: Platform Engineering; AI/ML Engineering
