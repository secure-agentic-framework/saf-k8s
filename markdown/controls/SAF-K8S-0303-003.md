# SAF-K8S-0303-003 - SELinux context assignment and multi-tenancy isolation

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.3 - Mandatory Access Controls
- Maturity: Practitioner
- Classification: baseline

## Statement

Clusters using SELinux-enabled nodes must assign SELinux contexts to containers with appropriate type enforcement labels and MCS (Multi- Category Security) labels for multi-tenancy isolation, and must not permit containers to run with unconfined or overly permissive SELinux types.

## Objective

Enforce mandatory access control through SELinux type enforcement and MCS category labels that provide kernel-level isolation between containers and between tenants, preventing unauthorized cross-container data access.

## Rationale

SELinux type enforcement restricts which kernel objects (files, sockets, processes) a container can access based on security labels rather than discretionary permissions. MCS labels provide an additional isolation layer by assigning unique category pairs to containers, preventing one container from accessing another's files even when they share the same SELinux type. Without SELinux context assignment, containers inherit permissive defaults that allow cross-container access through shared volumes and host filesystem paths.

## Scope

Applies to all clusters running on nodes with SELinux in enforcing mode (RHEL, CentOS, Fedora, and other distributions using SELinux as the default LSM). Does not apply to nodes using AppArmor as the primary LSM. MCS labels are particularly important for multi-tenant clusters sharing GPU or storage resources.

## Enforcement

- Primary Enforcement Point: Pod securityContext.seLinuxOptions (level, type, role, user) fields, enforced by policy engine rules validating SELinux context assignments at admission time.
- Implementation Surface: Pod securityContext.seLinuxOptions fields (level, type, role, user), SELinux MCS (Multi-Category Security) label allocation per tenant namespace, SELinux type enforcement labels (container_t), policy engine rules validating SELinux context assignments at admission, node-level SELinux enforcing mode configuration (RHEL, CentOS, Fedora, OpenShift), getenforce verification tooling, SELinux AVC denial log monitoring, and /proc/self/attr/current label inspection for running containers.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
