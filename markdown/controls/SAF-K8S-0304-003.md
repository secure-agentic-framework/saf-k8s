# SAF-K8S-0304-003 - QoS class assignment for workload stability

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.4 - Secure Defaults and Resource Constraints
- Maturity: Foundational
- Classification: baseline

## Statement

Security-critical and stateful workloads must be configured with Guaranteed QoS class (requests equal to limits for all resource types), and organizations must document the security implications of BestEffort and Burstable QoS classes, ensuring that no security-sensitive workload runs with BestEffort QoS.

## Objective

Ensure security-critical workloads receive predictable resource allocation and are protected from eviction during node pressure events, while preventing security-sensitive workloads from being assigned the lowest eviction priority.

## Rationale

Kubernetes evicts pods in QoS priority order: BestEffort first, then Burstable, then Guaranteed. Security-critical workloads (security monitoring agents, policy engines, log collectors) running with BestEffort or Burstable QoS can be evicted during node memory pressure, creating windows where security controls are absent. An attacker could intentionally trigger node pressure to evict security monitoring pods before launching an attack. Guaranteed QoS also prevents CPU throttling that could cause monitoring gaps.

## Scope

Applies to all workloads in production clusters. Security-critical workloads (admission controllers, security agents, audit log forwarders, monitoring DaemonSets) must use Guaranteed QoS. Application workloads may use Burstable QoS with documented acceptance of eviction risk. BestEffort QoS is prohibited for any workload handling sensitive data or performing security functions.

## Enforcement

- Primary Enforcement Point: Policy engine rules validating QoS class for workloads in security- critical namespaces, CI/CD pipeline checks verifying resource requests equal limits for designated workload classes.
- Implementation Surface: Pod spec resources.requests and resources.limits fields (setting requests equal to limits for Guaranteed QoS), policy engine rules validating QoS class for security-critical namespaces, CI/CD pipeline checks verifying Guaranteed QoS for designated workload classes, pod eviction monitoring dashboards, and Kubernetes scheduler QoS-based eviction priority configuration.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
