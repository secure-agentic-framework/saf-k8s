# SAF-K8S-0503-004 - Internal load balancer annotation enforcement

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.3 - Ingress, Egress, and DNS Hardening
- Maturity: Practitioner
- Classification: baseline

## Statement

Services of type LoadBalancer must use internal load balancer annotations by default unless explicitly approved for internet exposure through a documented security review process, with source IP restrictions (loadBalancerSourceRanges) applied to all approved external LoadBalancer services.

## Objective

Prevent unintended exposure of Kubernetes services to the public internet by enforcing internal load balancer annotations as the default for all LoadBalancer services.

## Rationale

A developer inadvertently creating a LoadBalancer service without internal annotations exposes the service to the public internet. Enforcing internal annotations by default ensures that internet exposure requires explicit opt-in through a security review process. Source IP restrictions provide defense in depth for approved external services. Admission-time enforcement of Service types is covered by SAF-K8S-0505-002.

## Scope

Applies to all production Kubernetes clusters using cloud provider LoadBalancer services. Covers internal annotation enforcement and source IP restriction for approved external services. Service type admission restrictions (blocking LoadBalancer, NodePort, ExternalIP) are addressed by SAF-K8S-0505-002.

## Enforcement

- Primary Enforcement Point: Admission controller policies enforcing internal load balancer annotations on LoadBalancer services.
- Implementation Surface: Internal load balancer annotations (AWS service.beta.kubernetes.io/aws-load-balancer-internal, GCP networking.gke.io/load-balancer-type: Internal, Azure service.beta.kubernetes.io/azure-load-balancer-internal), admission controller policies (Kyverno, OPA Gatekeeper) enforcing internal annotations by default, Service spec loadBalancerSourceRanges for source IP restrictions on approved external services, and kubectl-based audit scripts for externally exposed services.
- Owner Primary: Platform Engineering
- Owner Supporting: Network Engineering; Security Engineering
