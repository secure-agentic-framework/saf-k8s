# SAF-K8S-0803-001 - NVLink and NVSwitch traffic isolation for multi-GPU training

- Domain: D08 - GPU, Accelerator, and Confidential Computing
- Knowledge Area: 8.3 - High-Performance Interconnect Security
- Maturity: Advanced
- Classification: ai-specific

## Statement

Multi-GPU training environments using NVLink and NVSwitch must enforce traffic isolation between tenants, restrict NVSwitch topology information exposure, and implement controls to prevent cross-tenant data leakage over high-bandwidth GPU-to-GPU interconnects.

## Objective

Ensure that high-bandwidth NVLink and NVSwitch interconnects used for distributed training do not become a channel for cross-tenant data exfiltration or topology reconnaissance in shared GPU infrastructure.

## Rationale

NVLink provides up to 900 GB/s bandwidth between GPUs, and NVSwitch enables all-to-all GPU communication within a node. In multi-tenant environments, training jobs from different tenants may share NVSwitch fabric, enabling a malicious workload to sniff or inject traffic on the interconnect. NVSwitch topology information can reveal the physical GPU layout, enabling targeted scheduling attacks to co-locate with victim workloads on the same NVSwitch domain.

## Scope

Applies to Kubernetes clusters with multi-GPU nodes connected via NVLink or NVSwitch (DGX systems, HGX baseboards, multi-GPU servers). Single-tenant dedicated GPU servers with no shared NVSwitch fabric may document reduced requirements. Cloud GPU instances with provider-managed NVLink topology must verify isolation through provider attestation.

## Enforcement

- Primary Enforcement Point: GPU Operator NVSwitch configuration, NCCL environment variable controls for communication topology, pod scheduling constraints for NVSwitch domain isolation, and network policy equivalents for GPU interconnects.
- Implementation Surface: NVSwitch configuration and fabric partitioning on DGX/HGX systems, NCCL environment variables (NCCL_P2P_DISABLE, NCCL_NET, NCCL_SOCKET_IFNAME) controlling GPU-to-GPU communication topology, MIG partitioning for NVLink isolation within GPUs, topology-aware pod scheduling constraints (node affinity, NVSwitch domain boundaries), CUDA_VISIBLE_DEVICES environment variable restrictions per container, GPU Feature Discovery topology label suppression, and NVIDIA driver logs monitoring for cuCtxEnablePeerAccess calls across pod boundaries.
- Owner Primary: GPU Infrastructure Engineering
- Owner Supporting: ML Platform Engineering; Network Engineering; Security Engineering
