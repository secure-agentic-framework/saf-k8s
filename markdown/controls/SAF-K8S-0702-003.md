# SAF-K8S-0702-003 - LimitRange enforcement for containers and pods

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.2 - Namespace Isolation and Multi-Tenancy
- Maturity: Foundational
- Classification: baseline

## Statement

Every namespace must have a LimitRange object that defines default resource requests, default resource limits, and maximum resource limits for containers and pods, ensuring that no container can be created without resource constraints and that no single container can request resources exceeding the namespace's defined maximums.

## Objective

Guarantee that every container in the cluster operates within defined resource boundaries, preventing unbounded resource consumption and ensuring fair scheduling.

## Rationale

Without LimitRange enforcement, containers can be submitted with no resource requests or limits, making them invisible to the scheduler's resource accounting and allowing them to consume unbounded CPU and memory. This leads to node resource exhaustion, OOM kills affecting co-located workloads, and unpredictable scheduling behavior. LimitRange provides a namespace-level safety net that complements ResourceQuota.

## Scope

Applies to all namespaces in all Kubernetes clusters. Every namespace must have at least one LimitRange defining default requests, default limits, and maximum limits for containers. System namespaces may have relaxed maximums with documented justification. Covers CPU, memory, and ephemeral storage resource types.

## Enforcement

- Primary Enforcement Point: LimitRange objects per namespace, enforced by the Kubernetes LimitRange admission controller at pod creation time.
- Implementation Surface: LimitRange objects specifying default, defaultRequest, max, and min values for Container and Pod resource types (cpu, memory, ephemeral-storage), admission control policies (Kyverno, OPA/Gatekeeper) for per-pod GPU guardrails where LimitRange cannot enforce extended resources, and audit tooling to verify running workload compliance after LimitRange modifications.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering
