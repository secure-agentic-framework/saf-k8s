# SAF-K8S-0502-005 - AI workload data path encryption in transit

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.2 - CNI Plugins and Pod Networking Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Data paths between compute nodes and storage backends used for AI training and inference workloads must be encrypted in transit, covering pod-to-pod training communication, compute-to-storage paths, model registry download paths, and inference serving traffic, with documented compensating controls where encryption is not supported and performance impact evaluated for high-bandwidth training paths.

## Objective

Protect AI training data and model weights in transit between compute nodes and storage backends by ensuring all AI-specific data paths use appropriate in-transit encryption.

## Rationale

AI workloads transfer large volumes of sensitive data between GPU nodes and storage backends including training datasets, model checkpoints, and gradient updates. Unencrypted data paths expose this data to interception. NCCL TCP pod-to-pod training traffic, NFS mounts for training datasets, object storage access, model registry pulls, and RDMA paths each require distinct protection or compensating controls because they traverse different protocol stacks.

## Scope

Applies to Kubernetes clusters hosting AI/ML training or inference workloads. Covers pod-to-pod training traffic, compute-to-storage paths, model registry download paths, inference endpoint traffic, and RDMA paths requiring compensating controls where encryption is not supported. Excludes network policy design for AI traffic patterns, which is covered by SAF-K8S-0502-008.

## Enforcement

- Primary Enforcement Point: CNI encryption configuration, storage backend in-transit encryption settings, model registry TLS enforcement, and documented RDMA compensating controls.
- Implementation Surface: CNI WireGuard encryption for NCCL TCP pod-to-pod training traffic, storage backend in-transit encryption such as NFS Kerberos krb5p and HTTPS-enforced object storage, model registry TLS verification, InfiniBand/RoCE fabric isolation and switch ACLs where encryption is unavailable, and performance benchmarks comparing encrypted and unencrypted AI data paths.
- Owner Primary: Network Engineering
- Owner Supporting: ML Platform Engineering; Storage Engineering; Security Engineering
