# SAF-K8S-0402-021 - Workload token explicit audience binding

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.2 - Service Accounts and Workload Identity
- Maturity: Foundational
- Classification: baseline

## Statement

Projected workload service account tokens must specify explicit audiences appropriate to the target Kubernetes API or external relying service, and clusters must validate that workload token configurations do not issue audience-agnostic tokens for federated or API access.

## Objective

Prevent replay of workload tokens against unintended relying parties by requiring explicit audience binding for issued service account tokens.

## Rationale

Audience binding limits where a stolen token can be replayed. Even when projected tokens are used, a token without the correct explicit audience can be accepted too broadly by relying services or reused against unintended APIs. Explicit audience control narrows token use to the intended trust boundary.

## Scope

Applies to all projected workload service account tokens used against the Kubernetes API server or external relying services such as cloud workload identity endpoints.

## Enforcement

- Primary Enforcement Point: Projected token audience configuration validation and relying-service workload identity checks.
- Implementation Surface: Projected ServiceAccountToken volume audience settings, TokenRequest audience parameters, and validation evidence showing relying services accept only intended audience-bound tokens.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development
