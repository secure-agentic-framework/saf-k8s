# SAF-K8S-0102-014 - etcd certificate rotation coverage and execution

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Practitioner
- Classification: baseline

## Statement

Production Kubernetes clusters must implement automated rotation of etcd server certificates, peer certificates, and API server-to-etcd client certificates before expiration, with operational procedures or automation that ensure each certificate class is renewed and deployed without omission.

## Objective

Ensure that all etcd-related certificate classes are rotated before expiration so trust paths remain intact and expired credentials do not disrupt etcd availability.

## Rationale

Certificate failures commonly occur when one certificate class is excluded from the rotation path or renewal automation is incomplete. This control isolates certificate coverage and execution so teams can verify that server, peer, and client certificates are all renewed and deployed on schedule.

## Scope

Applies to all self-managed Kubernetes clusters using etcd. Covers etcd server, peer, and API server client certificates and the automation or operational procedures used to renew and deploy them.

## Enforcement

- Primary Enforcement Point: Certificate Rotation Workflow
- Implementation Surface: kubeadm or PKI automation configuration, cert-manager or equivalent issuance workflows, etcd manifest certificate references, and rotation execution records.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering
