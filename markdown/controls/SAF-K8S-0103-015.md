# SAF-K8S-0103-015 - Scheduler extender authentication and custom scheduler approval

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.3 - Controller-Manager, Scheduler, and Cloud Controller Security
- Maturity: Practitioner
- Classification: baseline

## Statement

Production Kubernetes clusters must require authentication and authorization for all scheduler extenders and webhook endpoints, and must permit custom schedulers only after security review and approval.

## Objective

Prevent unauthorized or untrusted scheduler extensions from influencing workload placement decisions without authenticated integration and explicit governance.

## Rationale

Unauthenticated scheduler extenders allow external parties to influence placement decisions, while unreviewed custom schedulers can bypass security-aware scheduling constraints entirely. Requiring authenticated extender integration and explicit approval for custom schedulers preserves trust in placement decisions while still allowing platform-specific scheduling extensions.

## Scope

Applies to all production clusters. Covers scheduler extender webhook endpoints, extender TLS and auth configuration, and governance for custom scheduler deployments including third-party GPU-oriented schedulers.

## Enforcement

- Primary Enforcement Point: Scheduler extension authentication and governance
- Implementation Surface: Scheduler extender webhook TLS and authentication configuration, admission or deployment approval records for custom schedulers, and periodic review of deployed scheduler extensions.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Platform teams
