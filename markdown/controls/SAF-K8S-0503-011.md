# SAF-K8S-0503-011 - External traffic policy mode selection and tradeoff governance

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.3 - Ingress, Egress, and DNS Hardening
- Maturity: Practitioner
- Classification: baseline

## Statement

Services exposed to external traffic must use externalTrafficPolicy settings appropriate for their security and availability requirements, and Local versus Cluster routing decisions must be documented with the associated tradeoffs.

## Objective

Ensure external traffic routing mode is chosen deliberately per service with documented security and availability tradeoffs.

## Rationale

Choosing Local versus Cluster externalTrafficPolicy affects availability and observability characteristics. Separating mode-selection governance from source IP preservation makes the routing decision independently auditable from the client identity handling requirement.

## Scope

Applies to all Services of type LoadBalancer or NodePort in production clusters that receive external traffic. The choice between Local and Cluster externalTrafficPolicy must be documented per service with security and availability justification.

## Enforcement

- Primary Enforcement Point: Kubernetes Service externalTrafficPolicy mode governance.
- Implementation Surface: Kubernetes Service spec externalTrafficPolicy field (Local vs. Cluster mode), pod topology and scheduling constraints for Local policy availability, and per-service documentation of traffic policy decisions with security and availability tradeoffs.
- Owner Primary: Network Engineering
- Owner Supporting: Application Development; Security Engineering
