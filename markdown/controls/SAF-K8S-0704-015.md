# SAF-K8S-0704-015 - Cloud IAM to Kubernetes RBAC entitlement mapping definition

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.4 - Cloud Provider Security Integration
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must establish and maintain a documented mapping between cloud provider IAM identities and Kubernetes RBAC bindings for managed Kubernetes services so intended Kubernetes permissions remain explicit and traceable.

## Objective

Ensure cloud IAM identities are intentionally translated into Kubernetes permissions through an explicit, documented entitlement mapping rather than implicit or undocumented access paths.

## Rationale

Managed Kubernetes environments bridge cloud IAM and Kubernetes RBAC, creating a two-layer authorization model. Without a documented entitlement mapping, teams cannot reliably determine which cloud identities resolve to which Kubernetes permissions, and access paths become opaque to design, review, and investigation.

## Scope

Applies to managed Kubernetes services and any self-managed clusters using cloud IAM or external cloud identity for Kubernetes authentication. The mapping must cover administrative users, automation identities, CI/CD roles, monitoring roles, and any other cloud identity granted Kubernetes access.

## Enforcement

- Primary Enforcement Point: Cloud IAM identity definitions and Kubernetes authentication-to-RBAC mapping documentation.
- Implementation Surface: EKS access entries or aws-auth mappings, GKE IAM-to-cluster role bindings, AKS Entra ID or equivalent identity integration, and documented mapping matrices tying cloud identities to Kubernetes RBAC subjects and grants.
- Owner Primary: Cloud Platform Engineering
- Owner Supporting: Security Engineering; Platform Engineering
