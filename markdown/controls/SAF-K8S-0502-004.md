# SAF-K8S-0502-004 - CNI plugin hardening and lifecycle management

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.2 - CNI Plugins and Pod Networking Security
- Maturity: Practitioner
- Classification: baseline

## Statement

CNI plugins must be deployed using organization-approved hardened configuration baselines, included in the vulnerability management program with timely patching of security advisories, and upgraded following documented procedures that include rollback plans and network connectivity validation.

## Objective

Maintain the security posture of the foundational networking layer by hardening CNI configurations and managing CNI plugin vulnerabilities and upgrades.

## Rationale

The CNI plugin operates with elevated privileges on every node and controls all pod network traffic. A vulnerable or misconfigured CNI plugin can be exploited for container escape, network policy bypass, or cluster-wide denial of service. Without hardened baselines and timely patching, known CVEs in CNI plugins remain exploitable. Unplanned or untested upgrades risk network outages that affect all cluster workloads.

## Scope

Applies to all CNI plugin deployments across all clusters. Includes the CNI binary, CNI DaemonSet, and any associated CRDs or control-plane components. Managed Kubernetes environments where the CNI is provider-managed must document the provider's patching SLA and compensating monitoring.

## Enforcement

- Primary Enforcement Point: CNI DaemonSet configuration management, container image vulnerability scanning, and change management procedures for CNI upgrades.
- Implementation Surface: Helm chart deployments with pinned versions and locked values files for CNI plugins, container image vulnerability scanning of CNI images (pinned by digest), CNI DaemonSet RBAC configuration, Pod Security Admission (Privileged profile for CNI namespace), CNI upgrade runbooks with --atomic rollback, post-upgrade connectivity tests (cilium connectivity test), and CVE patch management records for CNI components.
- Owner Primary: Platform Engineering
- Owner Supporting: Network Engineering; Security Engineering; Change Management
