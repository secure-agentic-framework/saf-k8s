# SAF-K8S-0203-012 - Kernel module restriction and approved module whitelisting

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.3 - Host OS and Kernel Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes nodes must restrict kernel module loading by disabling unnecessary modules, blacklisting known-risky modules, and maintaining an approved whitelist of required modules for node functions such as GPU drivers or InfiniBand support.

## Objective

Prevent unauthorized or unnecessary kernel modules from expanding the host kernel attack surface.

## Rationale

Unrestricted kernel module loading allows post-exploitation installation of rootkits, sniffers, and other kernel-resident capabilities below container isolation boundaries. Restricting modules through blacklists and approved whitelists is independently auditable from kernel lockdown mode because it governs which modules are permitted to exist on the running node.

## Scope

Applies to all production cluster nodes regardless of boot verification support. Nodes requiring specific kernel modules (e.g., GPU drivers, InfiniBand) must whitelist only required modules and document exceptions. DKMS-signed modules must have their signing keys validated.

## Enforcement

- Primary Enforcement Point: Node image build pipeline, modprobe configuration, and approved module inventory for production node pools.
- Implementation Surface: Kernel module blacklisting in /etc/modprobe.d/, approved module whitelist validation, lsmod verification, and exception handling for required modules such as signed GPU drivers.
- Owner Primary: OS/Infrastructure
- Owner Supporting: Platform Engineering; Security Engineering
