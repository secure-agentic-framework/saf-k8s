# SAF-K8S-0701-001 - PersistentVolume and PersistentVolumeClaim access mode enforcement

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.1 - Persistent Storage Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must configure PersistentVolume and PersistentVolumeClaim access modes using the least-permissive option required by the workload (preferring ReadWriteOnce or ReadOnlyMany over ReadWriteMany), and must enforce access mode restrictions for volumes containing training datasets, model checkpoints, and feature stores to prevent unauthorized cross-pod data modification.

## Objective

Prevent unintended cross-pod write access to persistent storage holding AI training data, model artifacts, and feature stores by enforcing least-privilege access modes.

## Rationale

ReadWriteMany volumes allow any pod on any node to read and write data simultaneously. If training dataset or model checkpoint volumes are inadvertently configured with ReadWriteMany, a compromised or misbehaving pod can corrupt training data, tamper with model checkpoints, or exfiltrate feature store contents. Restricting access modes limits the blast radius of a single pod compromise.

## Scope

Applies to all Kubernetes clusters hosting AI workloads where PersistentVolumes store training datasets, model checkpoints, feature stores, or inference artifacts. Covers both dynamically provisioned and statically provisioned volumes. ReadWriteMany may be permitted with documented justification and compensating controls for distributed training frameworks that require shared filesystem access.

## Enforcement

- Primary Enforcement Point: StorageClass parameters, PersistentVolumeClaim specs, and admission control policies (e.g., OPA Gatekeeper or Kyverno) that reject PVCs requesting overly permissive access modes without approved exceptions.
- Implementation Surface: PersistentVolume and PersistentVolumeClaim spec.accessModes configuration (ReadWriteOnce, ReadOnlyMany, ReadWriteMany), StorageClass definitions, admission control policy engines (Kyverno ClusterPolicy, OPA/Gatekeeper ConstraintTemplates) enforcing PVC access mode restrictions, namespace allowlisting for ReadWriteMany, pod affinity rules, and NetworkPolicy on storage backend endpoints.
- Owner Primary: Platform Engineering
- Owner Supporting: ML Engineering; Security Engineering
