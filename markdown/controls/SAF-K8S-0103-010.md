# SAF-K8S-0103-010 - Leader election configuration and lease object RBAC

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.3 - Controller-Manager, Scheduler, and Cloud Controller Security
- Maturity: Practitioner
- Classification: baseline

## Statement

Production Kubernetes clusters must enable leader election for kube-controller-manager, kube-scheduler, and cloud-controller-manager, configure lease duration, renew deadline, and retry period parameters appropriate to cluster latency, and restrict RBAC permissions for leader election Lease objects so that only the designated control-plane service accounts may read or update their respective Lease resources.

## Objective

Protect leader election from disruption or manipulation by ensuring safe timing parameters and tightly scoped access to Lease objects.

## Rationale

Leader election is both a configuration-safety problem and an authorization problem. Incorrect timing values cause unnecessary failovers or stuck leaders, while overly broad Lease-object permissions allow unauthorized workloads to interfere with the active control-plane leader. This control isolates those leader-election-specific safeguards from broader HA topology requirements.

## Scope

Applies to production clusters running kube-controller-manager, kube-scheduler, and cloud-controller-manager with leader election. Covers --leader-elect flags, lease timing parameters, and RBAC on Lease objects in kube-system.

## Enforcement

- Primary Enforcement Point: Leader Election / RBAC
- Implementation Surface: Component flag configuration, Lease-object RBAC roles and bindings, kube-system Lease resources, and latency-aware parameter documentation.
- Owner Primary: Platform Engineering
- Owner Supporting: Site Reliability Engineering; Security Engineering
