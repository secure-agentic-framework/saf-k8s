# SAF-K8S-0404-003 - mTLS for service-to-service authentication

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.4 - Certificate Management
- Maturity: Practitioner
- Classification: baseline

## Statement

Clusters must implement mutual TLS (mTLS) for service-to-service authentication, ensuring that both the client and server present valid certificates during the TLS handshake, with certificates bootstrapped through an automated identity framework or service mesh.

## Objective

Establish cryptographic mutual authentication between services, preventing unauthorized services from communicating with cluster workloads and enabling identity-based authorization at the transport layer.

## Rationale

Standard TLS only authenticates the server to the client. Without mTLS, any pod in the cluster that can reach a service endpoint can communicate with it, regardless of identity. An attacker who gains execution in any pod can reach and interact with services that accept unauthenticated connections. mTLS ensures both parties present verified certificates, enabling transport-layer identity verification and restricting communication to authorized workloads only.

## Scope

Applies to all service-to-service communication within production clusters. Implementation may use a service mesh (Istio, Linkerd, Cilium) or standalone SPIFFE/SPIRE identity framework. Exceptions for performance-critical paths (e.g., high-throughput GPU training communication) must document compensating network segmentation controls.

## Enforcement

- Primary Enforcement Point: Service mesh mTLS mode configuration (STRICT mode), SPIFFE/SPIRE workload attestation and SVID issuance, or application-level TLS configuration with client certificate verification.
- Implementation Surface: Service mesh mTLS configuration (Istio PeerAuthentication STRICT mode, Linkerd automatic mTLS, Cilium mutual authentication), SPIFFE/SPIRE workload attestation agents and SVID-based mTLS, application-level TLS client certificate verification, cert-manager Certificate resources for service identities, and documented mTLS exemptions for high-throughput GPU training communication (NCCL) with compensating network segmentation controls.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development
