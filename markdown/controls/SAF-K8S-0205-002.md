# SAF-K8S-0205-002 - NodePort and HostPort restriction policies

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.5 - kube-proxy and Node Networking Security
- Maturity: Practitioner
- Classification: baseline

## Statement

Clusters must enforce admission policies that restrict the use of NodePort services and HostPort pod specifications to approved exceptions only, preventing workloads from exposing services directly on node IP addresses without explicit authorization and security review.

## Objective

Limit node-level network exposure by preventing workloads from binding directly to host network ports, which bypasses ingress controls, load balancer security features, and network policy enforcement.

## Rationale

NodePort services expose a port on every cluster node's external IP address, creating a cluster-wide network entry point that bypasses ingress controller WAF rules, TLS termination, and rate limiting. HostPort specifications bind containers directly to the host network interface, bypassing pod-level network policies entirely. Both mechanisms create direct network paths to workloads that are invisible to standard ingress monitoring and difficult to audit across large clusters. Unrestricted use enables shadow entry points that persist even after intended ingress configurations are updated.

## Scope

Applies to all production cluster namespaces. System namespaces (kube-system, monitoring) may require NodePort or HostPort for infrastructure components (e.g., ingress controllers, node exporters) and must be explicitly whitelisted. Development clusters may relax this control with documented exceptions. Cloud environments should prefer LoadBalancer services or Ingress resources over NodePort exposure.

## Enforcement

- Primary Enforcement Point: Admission controllers (Pod Security Admission, OPA Gatekeeper, Kyverno) enforcing policies that deny NodePort services and HostPort specifications except in whitelisted namespaces.
- Implementation Surface: Admission policies (Kyverno, OPA Gatekeeper) validating Service type and pod hostPort fields, NodePort and HostPort denial rules, exemption registration process, and infrastructure component exception documentation (ingress controllers, node exporters).
- Owner Primary: Platform Engineering
- Owner Supporting: Network Engineering; Security Engineering
