# SAF-K8S-0202-004 - Runtime socket mount prevention

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.2 - Container Runtime Security
- Maturity: Foundational
- Classification: baseline

## Statement

Container workloads must not mount the container runtime socket into pod containers, enforced through admission controllers that block hostPath volume mounts to runtime socket paths, with CI/CD build pods using rootless or daemonless build tools instead of runtime socket mounting.

## Objective

Prevent containers from gaining direct access to the runtime control plane by blocking runtime socket mounts at the admission layer.

## Rationale

The container runtime socket (e.g., /run/containerd/containerd.sock) provides full control over container lifecycle operations including creation, exec, and image pulling. Mounting this socket inside a pod grants that container full control over all containers on the node, enabling privilege escalation to host root. Admission-layer blocking prevents this attack vector before pods are scheduled.

## Scope

Applies to all workloads on production cluster nodes. CI/CD build pods that require container image building must use rootless or daemonless build tools (kaniko, buildah) rather than runtime socket mounting. Exceptions for runtime socket access require documented risk acceptance and compensating monitoring controls.

## Enforcement

- Primary Enforcement Point: Admission controllers (e.g., Pod Security Admission, OPA Gatekeeper, Kyverno) blocking hostPath mounts to runtime socket paths.
- Implementation Surface: Admission controllers (Pod Security Admission, OPA Gatekeeper, Kyverno) blocking hostPath mounts for runtime sockets, CI/CD manifest scanning, and rootless build tools (kaniko, buildah, BuildKit).
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development
