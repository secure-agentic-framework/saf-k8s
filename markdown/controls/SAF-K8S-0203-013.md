# SAF-K8S-0203-013 - Kernel lockdown mode enforcement for runtime integrity

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.3 - Host OS and Kernel Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes nodes must enable kernel lockdown mode where supported so runtime loading of unsigned or unauthorized kernel code and direct kernel modification interfaces are restricted after boot.

## Objective

Reduce post-boot kernel tampering opportunities through kernel lockdown enforcement where supported.

## Rationale

Kernel lockdown mode restricts even privileged processes from loading unsigned modules or modifying kernel memory after boot, providing defense-in-depth when an attacker gains root on the node. This is distinct from module blacklisting because it enforces runtime integrity boundaries inside the kernel itself.

## Scope

Applies to all production cluster nodes regardless of boot verification support. Nodes requiring specific kernel modules (e.g., GPU drivers, InfiniBand) must whitelist only required modules and document exceptions. DKMS-signed modules must have their signing keys validated.

## Enforcement

- Primary Enforcement Point: Node boot and runtime kernel integrity settings for supported platforms and node pools.
- Implementation Surface: Kernel lockdown mode configuration and validation via runtime securityfs status checks, plus supported-platform exception handling.
- Owner Primary: OS/Infrastructure
- Owner Supporting: Platform Engineering; Security Engineering
