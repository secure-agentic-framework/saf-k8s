# SAF-K8S-0905-035 - Automatic canary rollback on error-rate and latency degradation

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.5 - AI Supply Chain and Model Lifecycle
- Maturity: Advanced
- Classification: ai-specific

## Statement

Canary and A/B deployment infrastructure must automatically roll back candidate model versions when defined error-rate, latency, or related degradation thresholds are exceeded during staged rollout.

## Objective

Limit the blast radius of a failed model rollout by automatically reverting to the stable version when health degradation is detected.

## Rationale

Even when versions are isolated and traffic splitting is correct, a candidate model can still degrade service quality or safety. Automatic rollback shortens exposure time by reverting to the stable version as soon as predefined health thresholds indicate an unsafe rollout.

## Scope

Applies to all multi-version model serving on Kubernetes, including canary deployments, A/B testing, and blue-green deployments. Covers degradation thresholds, rollback triggers, and automatic reversion behavior. Excludes promotion gate policy and approval workflow controls, which remain on adjacent promotion controls.

## Enforcement

- Primary Enforcement Point: Rollout controller health checks, analysis configuration, and rollback process.
- Implementation Surface: Automatic rollback triggers on error rate, latency, or related health degradation, rollback event logs, and controller policies proving failed candidate versions revert without manual delay.
- Owner Primary: ML Platform Engineering
- Owner Supporting: DevOps Engineering; Security Engineering
