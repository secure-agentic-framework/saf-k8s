# SAF-K8S-0904-007 - Scheduled feature computation job hardening

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.4 - AI Pipeline Orchestration and Experimentation Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Kubernetes CronJobs and batch Jobs used for scheduled feature computation, data refresh, or retraining preparation must be hardened for secure execution with least privilege, bounded runtime, and change control over job definitions.

## Objective

Ensure that scheduled ML data processing jobs execute securely so recurring feature computation and refresh tasks cannot be abused to exfiltrate data, run unauthorized code, or persist malicious changes.

## Rationale

Scheduled ML jobs often run unattended with broad data access and are easy to overlook once the initial pipeline is deployed. If CronJobs or batch Jobs are not hardened, an attacker can abuse recurring execution to inject malicious logic, exfiltrate data, or establish durable persistence through modified job definitions.

## Scope

Applies to all Kubernetes CronJobs and batch Jobs performing scheduled feature computation, data refresh, scheduled preprocessing, or retraining preparation for ML workloads. Covers pod security context, service account scope, execution bounds, and governance of scheduled job definitions.

## Enforcement

- Primary Enforcement Point: Kubernetes CronJob and Job specifications, admission policies for scheduled workloads, and RBAC controlling job creation and modification.
- Implementation Surface: CronJob and Job specifications with security context hardening (readOnlyRootFilesystem, non-root, dropped capabilities), dedicated service accounts with minimum data access, activeDeadlineSeconds and concurrency policy settings, RBAC restricting CronJob creation and modification, and change monitoring for CronJob definitions and schedules.
- Owner Primary: Data Engineering
- Owner Supporting: ML Platform Engineering; Security Engineering
