# SAF-K8S-0805-004 - GPU side-channel attack awareness and mitigation

- Domain: D08 - GPU, Accelerator, and Confidential Computing
- Knowledge Area: 8.5 - GPU Workload Auditing and Monitoring
- Maturity: Advanced
- Classification: ai-specific

## Statement

Organizations operating multi-tenant GPU infrastructure must assess side-channel risks including GPU shared memory timing attacks, memory access pattern leakage, and cross-tenant information disclosure, and must implement mitigation measures appropriate to the sensitivity of co-located workloads.

## Objective

Reduce the risk of information leakage through GPU hardware side channels in multi-tenant environments by ensuring organizations understand, assess, and mitigate GPU-specific side-channel attack vectors.

## Rationale

GPUs share hardware resources (L2 cache, memory controllers, shared memory banks) between concurrent workloads, creating side channels through which a malicious workload can infer information about co-located tenants. Timing variations in shared memory access reveal memory access patterns that can leak model architecture details or training data characteristics. Cache-based side channels on GPUs are less studied than CPU equivalents, meaning organizations may underestimate the risk. Without awareness and assessment, sensitive AI workloads may be unknowingly co-located with adversarial workloads capable of exploiting these channels.

## Scope

Applies to Kubernetes clusters where multiple tenants or security domains share GPU hardware, including time-sliced, MPS, and vGPU configurations. MIG-partitioned workloads have reduced but not eliminated side-channel risk. Single-tenant dedicated GPU pools with physical isolation may document reduced requirements. This control requires periodic reassessment as GPU side-channel research evolves.

## Enforcement

- Primary Enforcement Point: Workload co-location policies, GPU sharing mode selection (MIG preferred over time-slicing for sensitive workloads), scheduling anti-affinity rules for sensitive workloads, and periodic side-channel risk assessment process.
- Implementation Surface: Workload co-location policies restricting cross-trust-domain GPU sharing, GPU sharing mode selection guidance (MIG preferred for medium/high sensitivity, dedicated nodes for high sensitivity), scheduling anti-affinity rules for sensitive AI workloads, periodic GPU side-channel risk assessment process covering shared memory timing attacks, memory access pattern leakage, power analysis, and GPU scheduler observation vectors, workload sensitivity classification integrated with GPU placement decisions, and documented risk acceptance records for residual side-channel exposure in shared GPU environments.
- Owner Primary: Security Engineering
- Owner Supporting: ML Platform Engineering; GPU Infrastructure Engineering; Risk Management
