# SAF-K8S-0204-001 - Runtime security tool deployment for syscall and network monitoring

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.4 - Runtime Threat Detection
- Maturity: Practitioner
- Classification: baseline

## Statement

Production clusters must deploy at least one runtime security monitoring tool (e.g., Falco, Tetragon, KubeArmor) on all nodes to provide real-time syscall monitoring, network connection tracking, and file access observation, with alerts forwarded to the centralized security monitoring platform.

## Objective

Establish continuous runtime visibility into container and host behavior at the syscall level, enabling detection of malicious activity that bypasses application-layer and Kubernetes API-layer controls.

## Rationale

Kubernetes API audit logs and application-level monitoring do not capture host-level and container-level activities such as unexpected process execution, privilege escalation syscalls, or anomalous network connections. Without runtime security monitoring, container escape attempts, cryptomining, reverse shells, and lateral movement via host networking go undetected until impact becomes visible. Runtime tools operating at the kernel level (eBPF, kernel modules) provide the deepest detection capability available.

## Scope

Applies to all production cluster nodes, including GPU nodes and nodes hosting AI inference workloads. At least one runtime security tool must be deployed as a DaemonSet or host-level agent on every schedulable node. Managed Kubernetes services may satisfy this control through provider-native runtime threat detection services if they provide equivalent syscall-level visibility.

## Enforcement

- Primary Enforcement Point: Cluster bootstrapping automation deploying runtime security DaemonSets, with node affinity rules ensuring coverage on all node pools.
- Implementation Surface: Runtime security tool DaemonSet deployment (Falco, Tetragon, KubeArmor), node scheduling affinity ensuring coverage of GPU nodes, syscall and network monitoring configuration, alert forwarding to SIEM, and Kubernetes context enrichment settings.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; Security Operations (SOC)
