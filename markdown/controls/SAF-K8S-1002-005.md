# SAF-K8S-1002-005 - Metric endpoint authorization and RBAC

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.2 - Monitoring, Metrics, and Tracing
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must enforce authorization controls on Kubernetes metrics APIs (metrics.k8s.io, custom.metrics.k8s.io, external.metrics.k8s.io) and metric exporters, restricting read access to monitoring infrastructure service accounts and authorized platform operators.

## Objective

Ensure that authenticated principals can only access metric data appropriate to their role, preventing workload tenants and developer accounts from reading cluster-wide or cross-namespace metric data.

## Rationale

Authentication alone does not prevent an authenticated but over-privileged account from querying metrics it should not see. RBAC on the metrics aggregation layer ensures that HPA controllers access only their namespace metrics, workload service accounts cannot enumerate cluster-wide resource data, and audit logs capture all metrics API access for review.

## Scope

Applies to the Kubernetes metrics API aggregation layer (metrics.k8s.io, custom.metrics.k8s.io, external.metrics.k8s.io) and to Prometheus RBAC for Kubernetes service discovery.

## Enforcement

- Primary Enforcement Point: Kubernetes RBAC, including ClusterRole bindings for system:metrics-reader, custom metrics API roles, and Prometheus service account RBAC for service discovery.
- Implementation Surface: metrics-server RBAC (system:metrics-reader ClusterRole binding restrictions to monitoring namespace service accounts), custom metrics API RBAC (custom.metrics.k8s.io and external.metrics.k8s.io restricted to HPA and KEDA controllers), Prometheus ServiceAccount RBAC scoped to read-only access on endpoints/pods/services/nodes.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Operations,AI/ML Engineering
