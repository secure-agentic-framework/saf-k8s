# SAF-K8S-0403-002 - External secrets management integration

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.3 - Secrets Management
- Maturity: Foundational
- Classification: baseline

## Statement

Organizations must integrate Kubernetes workloads with external secrets management systems (External Secrets Operator, Vault CSI Provider, or cloud-native secret stores) to centralize secret storage, enable automatic synchronization, and enforce rotation policies outside the Kubernetes control plane.

## Objective

Centralize secrets in a dedicated secrets management platform with stronger access controls, audit logging, and rotation capabilities than native Kubernetes Secrets provide, while ensuring workloads receive up-to-date secret values automatically.

## Rationale

Native Kubernetes Secrets lack built-in rotation, fine-grained audit logging, and centralized management across multiple clusters. Storing secrets exclusively in Kubernetes creates cluster-scoped blast radius and makes cross-cluster secret management difficult. External secrets management platforms provide centralized lifecycle management, automatic rotation, detailed access audit trails, and integration with enterprise compliance workflows.

## Scope

Applies to all production clusters and workloads that consume secrets. Covers database credentials, API keys, TLS certificates, and any other sensitive configuration. Organizations using multiple clusters must ensure consistent external secrets integration across all clusters. Exceptions for bootstrapping secrets (e.g., initial Vault authentication) must be documented.

## Enforcement

- Primary Enforcement Point: External Secrets Operator or Vault CSI Provider deployment configuration, SecretStore and ExternalSecret custom resource definitions, and admission policies restricting direct creation of native Kubernetes Secrets.
- Implementation Surface: External Secrets Operator (ESO) with SecretStore and ExternalSecret custom resources, Vault CSI Provider for direct pod volume mounting, Vault Agent sidecar injector, cloud-native secret stores (AWS Secrets Manager, GCP Secret Manager, Azure Key Vault), synchronization interval and event-driven refresh configurations, and admission policies restricting direct creation of native Kubernetes Secret objects.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; Application Development
