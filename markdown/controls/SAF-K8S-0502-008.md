# SAF-K8S-0502-008 - Network policy design for AI-specific traffic patterns

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.2 - CNI Plugins and Pod Networking Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Network policies for Kubernetes clusters hosting AI workloads must be designed to accommodate AI-specific traffic patterns including NCCL inter-node training communication with pinned port ranges scoped to training job labels, model registry download paths restricted to approved registries, and inference serving traffic, preventing both inadvertent blocking of legitimate AI communication and over-permissive rules that create lateral movement paths.

## Objective

Ensure network policies correctly handle the unique traffic patterns of distributed AI workloads without blocking legitimate communication or creating excessive lateral movement exposure between unrelated workloads.

## Rationale

AI workloads generate non-standard traffic patterns that standard network policies are not designed to handle. NCCL all-reduce operations require direct pod-to-pod communication across nodes on high-port ranges; without job-label-scoped policies, this traffic is either blocked, causing training failures, or permitted too broadly, creating lateral movement paths between training jobs. Model download paths require registry-specific egress rules to prevent exfiltration via alternate endpoints. Inference serving traffic requires differentiated policies by role and criticality. These trust-boundary decisions are independent of in-transit encryption.

## Scope

Applies to Kubernetes clusters hosting AI/ML training or inference workloads. Covers NCCL inter-node training traffic, model registry egress policies, inference endpoint ingress and egress policies, and parameter server communication patterns. Excludes in-transit encryption configuration and RDMA fabric compensating controls, which remain on SAF-K8S-0502-005.

## Enforcement

- Primary Enforcement Point: Kubernetes NetworkPolicy manifests with training-job label selectors, pinned NCCL port ranges, and explicit registry / inference path allowlists.
- Implementation Surface: Kubernetes NetworkPolicy rules scoped to training job labels with pinned NCCL port ranges, default-deny egress policies for AI namespaces with explicit allowlists for model registries and object storage endpoints, inference namespace ingress policies permitting only approved gateway or load balancer sources, and parameter-server communication policies scoped to job-specific labels to prevent cross-job access.
- Owner Primary: Network Engineering
- Owner Supporting: ML Platform Engineering; Security Engineering
