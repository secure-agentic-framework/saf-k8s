# SAF-K8S-0704-008 - Cloud-to-Kubernetes event correlation for incident investigation

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.4 - Cloud Provider Security Integration
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must implement and maintain correlation logic that links cloud-level actions to their Kubernetes-level effects so investigators can reconstruct end-to-end attack chains, administrative changes, and security-relevant operational events in managed Kubernetes environments.

## Objective

Enable end-to-end incident investigation by correlating cloud infrastructure actions with Kubernetes control plane and workload effects.

## Rationale

Cloud and Kubernetes audit data only becomes operationally useful when investigators can connect cause and effect across the shared-responsibility boundary. Correlation rules close that gap by linking cloud IAM, infrastructure, and service-provider events to the Kubernetes actions and workload impacts they produce.

## Scope

Applies to all managed Kubernetes clusters on cloud providers (EKS, GKE, AKS). Correlation logic must cover at minimum IAM role changes affecting Kubernetes access, node provisioning events, load balancer modifications, storage operations, and other cloud actions with Kubernetes impact.

## Enforcement

- Primary Enforcement Point: Correlation rule design, identity linking across cloud IAM and Kubernetes audit records, investigation workflow validation, and analytic tuning.
- Implementation Surface: Correlation rules linking cloud IAM identities to Kubernetes user identities via IAM-to-RBAC mapping, cloud request identifiers to Kubernetes audit metadata, timestamp-based event windows, and investigation playbooks for high-priority scenarios such as IAM role assumption followed by cluster-admin activity or infrastructure changes followed by workload impact.
- Owner Primary: Security Engineering
- Owner Supporting: Cloud Platform Engineering; Platform Engineering; SOC
