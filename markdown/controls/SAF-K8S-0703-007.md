# SAF-K8S-0703-007 - Fair-share GPU queue management for multi-tenant clusters

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.3 - Resource Governance and Priority
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must deploy queue management (Kueue, Volcano, or equivalent) for fair-share GPU scheduling across tenants in clusters shared by multiple teams submitting GPU workloads.

## Objective

Enable fair and predictable access to GPU resources across teams by using queue-based scheduling that prevents resource hoarding and GPU starvation.

## Rationale

Without queue management, GPU scheduling in multi-tenant clusters relies solely on Kubernetes native scheduling, which cannot enforce fair-share policies, gang scheduling for distributed training, or quota-aware queuing. This leads to GPU starvation for some teams and resource hoarding by others. Kueue and Volcano provide queue abstractions that enable fair-share scheduling, resource borrowing, and preemption policies that are not achievable with PriorityClass alone.

## Scope

Applies to all Kubernetes clusters shared across multiple teams submitting GPU workloads. Single-tenant dedicated clusters may use simplified scheduling with documented justification. The queue management system must support fair-share scheduling, resource borrowing policies, and gang scheduling for distributed training jobs.

## Enforcement

- Primary Enforcement Point: Kueue or Volcano queue configuration with fair-share policies and borrowing limits for GPU resources.
- Implementation Surface: Kueue ClusterQueues with borrowing limits and fair-share policies, Volcano queue management and gang scheduling configuration, resource flavor definitions for GPU types, and cohort-level fair-sharing policies across tenants.
- Owner Primary: Platform Engineering
- Owner Supporting: ML Engineering; FinOps
