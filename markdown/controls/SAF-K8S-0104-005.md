# SAF-K8S-0104-005 - Control plane configuration file permissions

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.4 - CIS Benchmarks and Patch Management
- Maturity: Foundational
- Classification: baseline

## Statement

Control plane nodes must enforce restrictive ownership and permissions on static pod manifests, kubeconfig files, certificate key material, and the etcd data directory, following the CIS Kubernetes Benchmark file permission requirements (section 1.1), and must verify compliance on node provisioning and on a recurring schedule.

## Objective

Prevent unauthorized read or modification of control plane configuration, credentials, and cluster state by enforcing file-system-level access controls on control plane nodes.

## Rationale

Control plane configuration files contain security-sensitive content: static pod manifests reveal API server flags, volume mounts, and host port bindings; kubeconfig files contain client credentials; private keys enable impersonation of cluster components; and the etcd data directory holds all cluster state. Overly permissive file ownership or modes allow local attackers or compromised processes on a control plane node to read credentials, modify component behavior, or tamper with cluster state.

## Scope

Applies to all self-managed control plane nodes. Covers static pod manifests, component kubeconfig files, PKI key material, and the etcd data directory. For managed Kubernetes services, this control is provider-managed and should be documented as such in the shared responsibility model.

## Enforcement

- Primary Enforcement Point: Node Configuration
- Implementation Surface: Node provisioning automation (cloud-init, Ansible, Terraform), node image build pipelines, and periodic configuration management scans.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering
