# SAF-K8S-0606-029 - GitOps deployed package and version metadata retention

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.6 - CI/CD and GitOps Pipeline Security
- Maturity: Practitioner
- Classification: baseline

## Statement

GitOps deployment processes must preserve release metadata for deployed packages and version identifiers so auditors and responders can determine exactly which package builds and versions were released to each environment.

## Objective

Ensure GitOps-managed deployments retain package and version metadata sufficient to identify the exact released artifacts and package versions promoted into each environment.

## Rationale

Without preserved package and version metadata, teams cannot prove which package build or chart version was deployed, correlate deployed software to vulnerability disclosures, or support precise rollback and audit review. This is a distinct obligation from tracking configuration revision history.

## Scope

Applies to production and staging GitOps deployments and to package, chart, image, and release-version metadata needed to identify what software release was deployed.

## Enforcement

- Primary Enforcement Point: GitOps release records and retained package or version metadata.
- Implementation Surface: Helm release metadata, OCI artifact tags and digests, image version records, package manifests, and deployment history retained by GitOps controllers or release systems.
- Owner Primary: Platform Engineering
- Owner Supporting: DevSecOps / Release Engineering; Security Engineering
