# SAF-K8S-0702-005 - Label and annotation schema definition for AI workload classification

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.2 - Namespace Isolation and Multi-Tenancy
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must define a standardized labeling and annotation schema for all AI workloads that classifies workload type (training, inference, data preprocessing, evaluation), data sensitivity level, and model tier, and must document this schema to enable consistent identification, policy enforcement, scheduling, and auditability of AI workloads.

## Objective

Enable consistent identification, policy enforcement, scheduling, and auditability of AI workloads by requiring machine-readable classification metadata on every pod.

## Rationale

Without standardized workload classification labels, platform teams cannot apply differentiated security policies, resource quotas, scheduling rules, or network policies to AI workloads. Inference workloads may inadvertently run on training-optimized nodes, sensitive data processing may co-locate with untrusted workloads, and cost attribution becomes impossible. A documented schema is the foundation for all downstream policy enforcement.

## Scope

Applies to all pods, jobs, deployments, and statefulsets in namespaces designated for AI workloads. The labeling schema must cover at minimum: workload type (training, inference, preprocessing, evaluation, notebook), data sensitivity (public, internal, confidential, restricted), and model tier (experimental, staging, production). Non-AI workloads in shared clusters should carry a baseline classification label.

## Enforcement

- Primary Enforcement Point: Admission control policies (OPA Gatekeeper, Kyverno) that validate required labels on pod creation, and namespace-level label enforcement through mutating or validating webhooks.
- Implementation Surface: Kubernetes pod metadata.labels schema (ai.kais.io/workload-type, ai.kais.io/data-sensitivity, ai.kais.io/model-tier, ai.kais.io/team), admission control policy engines (Kyverno ClusterPolicy, OPA/Gatekeeper ConstraintTemplates) validating required label presence and value enumeration on pods in AI namespaces, label-based NetworkPolicy selectors, ResourceQuota scopeSelectors keyed on labels, and audit log filtering by classification labels.
- Owner Primary: Platform Engineering
- Owner Supporting: ML Engineering; Data Governance; Security Engineering
