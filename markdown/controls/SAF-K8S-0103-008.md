# SAF-K8S-0103-008 - Cloud controller-manager deployment isolation

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.3 - Controller-Manager, Scheduler, and Cloud Controller Security
- Maturity: Practitioner
- Classification: baseline

## Statement

Production Kubernetes clusters using a cloud controller-manager must deploy it separately from the core kube-controller-manager so that cloud-provider-specific controller execution, privileges, and failure modes are isolated from the core control-plane controller process.

## Objective

Limit the blast radius of a cloud controller-manager compromise or failure by separating it from the core controller-manager execution boundary.

## Rationale

Co-deploying the cloud controller-manager with the core controller-manager collapses trust boundaries between cloud-facing automation and core Kubernetes reconciliation logic. This control isolates the deployment and process-boundary requirement so teams can independently verify component separation from cloud IAM and credential lifecycle concerns.

## Scope

Applies to production clusters with cloud controller-manager integrations. Covers separate deployment topology, process boundary, and operational separation from kube-controller-manager.

## Enforcement

- Primary Enforcement Point: Component Isolation
- Implementation Surface: Cloud controller-manager deployment manifests, static pod or deployment topology, and component process inventory showing separate execution from kube-controller-manager.
- Owner Primary: Platform Engineering
- Owner Supporting: Cloud Platform Engineering; Security Engineering
