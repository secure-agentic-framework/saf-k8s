# SAF-K8S-0301-003 - PodSecurityPolicy to PSA migration

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.1 - Pod Security Standards and Admission
- Maturity: Foundational
- Classification: baseline

## Statement

Organizations still operating legacy PodSecurityPolicy (PSP) must execute a documented migration plan to Pod Security Admission that includes gap analysis of existing PSP rules, mapping of PSP policies to PSS levels, dry-run validation in audit and warn modes, and phased namespace-by-namespace rollout before PSP removal.

## Objective

Eliminate reliance on the deprecated PodSecurityPolicy API while preserving equivalent or stronger workload security controls through a structured, validated migration to Pod Security Admission.

## Rationale

PodSecurityPolicy was removed in Kubernetes 1.25. Clusters that have not migrated to PSA or an equivalent policy engine operate without any pod-level security admission enforcement, leaving all workloads unconstrained. Unplanned migration can introduce gaps where previously enforced policies are silently lost, creating windows of vulnerability.

## Scope

Applies to any cluster that previously relied on PodSecurityPolicy or is upgrading from a Kubernetes version that supported PSP. Clusters already running PSA with validated policies are exempt. Migration plans must cover both application and system namespaces.

## Enforcement

- Primary Enforcement Point: Migration project governance, with validation through PSA audit and warn mode results before switching to enforce mode.
- Implementation Surface: PodSecurityPolicy API objects (legacy), Pod Security Admission namespace labels, policy engines (OPA Gatekeeper, Kyverno) for gap coverage of PSP features not present in PSA (per-field granularity, mutation), PSA audit and warn mode logs, migration project governance tooling, and phased namespace-by-namespace rollout automation.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
