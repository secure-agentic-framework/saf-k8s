# SAF-K8S-1007-005 - Change management for production AI model deployments

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.7 - Cluster Lifecycle and Asset Inventory
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must enforce change management procedures for production model deployments, including approval workflows, defined deployment windows, and tested rollback procedures.

## Objective

Ensure that AI model deployments follow controlled change processes to prevent introduction of compromised or inadequately tested models into production.

## Rationale

Uncontrolled model deployments can introduce compromised or inadequately tested models into production without security review. Without formal change management, there is no assurance that deployed models have passed security validation, and no tested rollback path exists if a deployment introduces issues.

## Scope

Applies to all production Kubernetes clusters hosting AI workloads. Change management covers model deployment, update, and rollback operations.

## Enforcement

- Primary Enforcement Point: Change management approval system and ML pipeline promotion gates.
- Implementation Surface: Model deployment approval workflows (promotion gates in ML pipelines requiring model owner and platform security team review), deployment window definitions for production inference services, InferenceService spec rollback procedures (revert to previous model version, health check verification, accuracy metric confirmation), and canary deployment configuration (traffic percentage routing, accuracy and latency metric monitoring).
- Owner Primary: Platform Engineering
- Owner Supporting: AI/ML Engineering; Security Operations
