# SAF-K8S-0703-013 - Topology-aware scheduling for GPU locality with blast-radius containment

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.3 - Resource Governance and Priority
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must implement topology-aware scheduling that places distributed training jobs on GPUs connected via high-speed interconnects (NVLink, NVSwitch) to maximize communication bandwidth, while spreading workloads across failure domains to limit blast radius.

## Objective

Optimize distributed AI training performance by scheduling GPU workloads with NVLink/NVSwitch locality awareness, while balancing placement to prevent single-point failure from taking down entire training jobs.

## Rationale

Without topology-aware scheduling, distributed training jobs may be placed on GPUs without high-speed interconnects (NVLink/NVSwitch), severely degrading performance and wasting expensive GPU time. However, concentrating all GPU workloads on a single node or failure domain creates a blast-radius risk.

## Scope

Applies to Kubernetes clusters running distributed multi-GPU training jobs. Topology Manager policies must be configured on GPU nodes. Node labels must indicate GPU interconnect topology (NVLink, NVSwitch connectivity).

## Enforcement

- Primary Enforcement Point: Topology Manager policy configuration on GPU nodes and topology spread constraints in pod specs for distributed training and inference workloads.
- Implementation Surface: Kubernetes Topology Manager policies (single-numa-node, best-effort), node labels indicating GPU topology (NVLink, NVSwitch connectivity), topology spread constraints for inference replica distribution across failure domains, and scheduler plugin configuration for GPU topology awareness.
- Owner Primary: Platform Engineering
- Owner Supporting: ML Engineering, SRE
