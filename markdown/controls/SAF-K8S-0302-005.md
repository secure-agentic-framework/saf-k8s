# SAF-K8S-0302-005 - AI workload security context hardening profiles

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.2 - Security Contexts and Capabilities
- Maturity: Foundational
- Classification: ai-specific

## Statement

AI workloads (inference servers, training jobs, data pipelines) must enforce read-only root filesystems, non-root execution, and workload- specific seccomp or AppArmor profiles tailored to GPU syscall patterns (GPU ioctl, shared memory operations, RDMA verbs), rather than running with unconfined or default profiles that permit unnecessary syscalls.

## Objective

Harden AI and ML workloads with security context configurations and mandatory access control profiles specifically tuned to the syscall patterns of GPU-accelerated workloads, closing attack surface without breaking GPU functionality.

## Rationale

AI workloads running inference servers or training jobs are frequently deployed with overly permissive security contexts because teams assume GPU access requires unrestricted privileges. In practice, GPU workloads use a well-defined set of syscalls (ioctl for GPU device interaction, shmget/mmap for shared memory, RDMA verbs for high-performance networking) that can be profiled and enforced. Unconfined profiles allow attackers who compromise an ML container to execute arbitrary syscalls including those needed for container escape.

## Scope

Applies to all AI/ML workloads in production clusters, including inference servers, training jobs, feature engineering pipelines, and model serving containers that use GPU or accelerator resources. Profiles must be validated against actual workload syscall behavior before enforcement.

## Enforcement

- Primary Enforcement Point: Security context fields in pod specifications, seccomp profile annotations or securityContext.seccompProfile, AppArmor annotations, enforced by PSA and policy engine admission rules.
- Implementation Surface: Pod and container securityContext fields (readOnlyRootFilesystem, runAsNonRoot, runAsUser, seccompProfile, capabilities.drop), Security Profiles Operator (SPO) for custom seccomp profile generation and deployment, AppArmor profile annotations (container.apparmor.security.beta.kubernetes.io or securityContext.appArmorProfile), GPU workload syscall profiling tools (strace, seccomp log mode), emptyDir and PVC volume mounts for model cache and scratch space, NVIDIA device plugin with CDI for zero-capability GPU access, and policy engine admission rules enforcing workload-class-specific security context templates for inference servers, training jobs, and data pipelines.
- Owner Primary: AI/ML Platform team
- Owner Supporting: Platform Engineering; Security Engineering
