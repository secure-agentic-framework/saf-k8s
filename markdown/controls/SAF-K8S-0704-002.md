# SAF-K8S-0704-002 - VPC and security group integration with Kubernetes network policies

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.4 - Cloud Provider Security Integration
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must configure VPC networking, security groups, and cloud firewalls to isolate Kubernetes node networks, restrict control plane endpoint access to authorized networks, and align cloud-level network controls with Kubernetes NetworkPolicy objects to create consistent, layered network security without gaps or conflicts.

## Objective

Create defense-in-depth network isolation by ensuring that cloud-level network controls and Kubernetes-level network policies enforce consistent security boundaries.

## Rationale

Cloud VPC security groups and Kubernetes NetworkPolicy operate at different layers of the network stack. If cloud security groups permit broad inter-node traffic while NetworkPolicy restricts pod-to-pod communication, an attacker who compromises a node can bypass pod-level network policies entirely. Conversely, if the API server control plane endpoint is publicly accessible, attackers can attempt credential-based attacks without needing network-level access. Misalignment between these layers creates exploitable gaps.

## Scope

Applies to all cloud-hosted Kubernetes clusters. Covers VPC subnet design for node networks, security group rules for inter-node and external traffic, control plane endpoint access restrictions (private endpoint, authorized networks), and alignment between cloud firewalls and Kubernetes NetworkPolicy. Must be reviewed when VPC topology or Kubernetes network configuration changes.

## Enforcement

- Primary Enforcement Point: Cloud VPC and security group configuration, API server endpoint access settings (private endpoint, authorized networks), and infrastructure-as-code templates that co-define cloud and Kubernetes network policies.
- Implementation Surface: VPC subnet configuration isolating Kubernetes nodes in dedicated subnets, cloud security groups and firewall rules for node-to-node and node-to-external traffic, managed Kubernetes API server private endpoint configuration and authorized CIDR restrictions, Kubernetes NetworkPolicy objects (CNI-enforced pod-level rules) aligned with VPC-level controls, egress security group rules permitting registry endpoints, cloud service APIs, and DNS, and layered defense-in-depth documentation mapping VPC coarse-grained controls to Kubernetes fine-grained pod-level policies.
- Owner Primary: Cloud Platform Engineering
- Owner Supporting: Network Engineering; Platform Engineering; Security Engineering
