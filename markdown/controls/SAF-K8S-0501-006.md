# SAF-K8S-0501-006 - Multi-cluster network segmentation for federated AI workloads

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.1 - Network Policies
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Multi-cluster AI deployments (federated training, distributed inference, cross-cluster model serving) must implement network segmentation controls that restrict cross-cluster traffic to explicitly authorized AI communication paths, enforce mutual authentication on all cross-cluster connections, and provide unified flow visibility across cluster boundaries.

## Objective

Extend network segmentation guarantees across cluster boundaries for multi-cluster AI workloads, ensuring that cross-cluster AI traffic is authorized, authenticated, encrypted, and auditable.

## Rationale

Single-cluster network policies do not extend across cluster boundaries. When AI workloads span multiple clusters -- federated training with participants in different clusters, multi-region inference routing, or hybrid on-prem/cloud training -- the absence of cross-cluster segmentation controls allows unauthorized inter-cluster communication, gradient exfiltration from federated learning participants, lateral movement through cross-cluster service discovery, and data residency violations when training data flows cross uncontrolled network paths.

## Scope

Applies to any Kubernetes deployment where AI workloads communicate across cluster boundaries, including federated learning with cross-cluster participants, multi-region inference serving, hybrid on-prem/cloud training, and cross-cluster model registry access. Covers multi-cluster service mesh, Kubernetes Multi-Cluster Services API, VPN/interconnect, and API gateway federation models.

## Enforcement

- Primary Enforcement Point: Multi-cluster service mesh policies (Cilium Cluster Mesh CiliumNetworkPolicy, Istio multi-cluster AuthorizationPolicy), gateway pod allowlists, cross-cluster ingress/egress controls, and unified flow logging infrastructure.
- Implementation Surface: Multi-cluster service mesh policies (Cilium Cluster Mesh CiliumNetworkPolicy with io.cilium.k8s.policy.cluster selectors, Istio multi-cluster AuthorizationPolicy with source principal matching), Kubernetes Multi-Cluster Services API (KEP-1645), VPN/interconnect gateway pods with mTLS allowlists and rate limiting, API gateway federation for cross-cluster inference routing, cross-cluster communication matrix documentation, Cilium Hubble multi-cluster relay for unified flow logging, VPC flow logs with cross-account aggregation, SPIFFE/SPIRE trust bundle federation for cross-cluster identity, and cross-cluster connectivity testing tools.
- Owner Primary: Network Engineering
- Owner Supporting: ML Platform Engineering; Security Operations
