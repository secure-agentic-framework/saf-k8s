# SAF-K8S-1002-006 - Sensitive metric redaction and access restriction

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.2 - Monitoring, Metrics, and Tracing
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must redact or restrict access to sensitive metrics that reveal GPU topology, training hyperparameters, or proprietary AI model behavior data, using Prometheus relabeling, recording rules, or namespace-scoped metric APIs to prevent exposure to unauthorized tenants.

## Objective

Prevent leakage of competitively sensitive or security-relevant AI infrastructure data through metric exposition, ensuring GPU topology, model hyperparameters, and inference performance data are accessible only to authorized teams.

## Rationale

Metric data can carry high-value intellectual property: GPU NVLink topology reveals hardware layout (competitive intelligence), training hyperparameters reveal model architecture choices (IP), and inference accuracy trends reveal model performance (business data).

## Scope

Applies to all metric labels, dimensions, and time series that expose GPU topology, training job internals, and inference serving internals. Covers both Prometheus-scraped metrics and custom metrics pushed via push gateway.

## Enforcement

- Primary Enforcement Point: Prometheus relabeling configuration (metric_relabel_configs) and recording rules that aggregate or drop sensitive label dimensions before storage.
- Implementation Surface: Prometheus metric_relabel_configs to drop or hash sensitive label values (GPU device UUIDs, NVLink topology labels), Prometheus recording rules to produce safe aggregated summaries from detailed training metrics, namespace-scoped metric APIs restricting training hyperparameter metrics.
- Owner Primary: AI/ML Engineering
- Owner Supporting: Security Operations,Platform Engineering
