# SAF-K8S-0604-007 - Third-party component security requirements documentation

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.4 - SBOMs and Vulnerability Intelligence
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must document security requirements for third-party software components admitted to the cluster, including container base images, Helm charts, operators, ML frameworks, and pre-trained models. Requirements must specify minimum expectations for vulnerability disclosure practices, SBOM availability, image signing, supported lifecycle and patching cadence, and license compliance. Requirements must be communicated to vendors and open-source project maintainers through procurement documents, supplier agreements, or published intake criteria. Admission control policies must enforce the verifiable subset of these requirements.

## Objective

Formalize and communicate the security expectations that third-party software components must meet before admission to the cluster, closing the gap between infrastructure-level enforcement and upstream supplier engagement.

## Rationale

SAF-K8S admission controls enforce technical requirements (signatures, scanning) at the cluster boundary, but without documented requirements communicated to suppliers, organizations cannot hold vendors accountable for missing SBOMs, unsigned images, or absent vulnerability disclosure practices. SSDF task PO.1.3 requires communication of requirements to third parties.

## Scope

All third-party software components consumed by the Kubernetes platform and AI workloads: container images, Helm charts, operators, CSI/CNI plugins, ML frameworks, pre-trained models, and pipeline components.

## Enforcement

- Primary Enforcement Point: admission-controller
- Implementation Surface: Procurement and vendor management documentation, supplier security questionnaires, published intake criteria for open-source components, admission controller policies enforcing SBOM presence and image signature requirements, model quarantine workflow intake checklists.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; ML Platform Engineering; Procurement

#### Framework Mapping Notes

The technical documentation of a high-risk AI system shall be drawn up before that system is placed on the market or put into service and shall be kept up to date. The technical documentation shall be drawn up in such a way as to demonstrate that the high-risk AI system complies with the requirements set out in this Section and to provide national competent authorities and notified bodies with the necessary information in a clear and comprehensive form to assess the compliance of the AI system with those requirements. It shall contain, at a minimum, the elements set out in Annex IV.

SAF-K8S supports technical documentation through ML-BOM generation (SAF-K8S-0604-002), SBOM generation (SAF-K8S-0604-001), model provenance tracking (D9 KA 9.5), and training data provenance (D9 KA 9.6). The organizational obligation to produce and maintain comprehensive technical documentation is out of scope; SAF-K8S provides the infrastructure artifacts that feed into that documentation.
