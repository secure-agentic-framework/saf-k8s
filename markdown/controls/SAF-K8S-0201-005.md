# SAF-K8S-0201-005 - Kubelet hostname override governance

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.1 - Kubelet and Node Configuration Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must govern the use of the kubelet --hostname-override flag by prohibiting it by default, requiring documented justification for any permitted use (e.g., cloud provider node-naming requirements), and verifying that any override value does not allow a kubelet to assume another node's identity or bypass Node authorization identity binding.

## Objective

Prevent kubelet identity spoofing through hostname override misconfiguration while preserving the ability to use the flag in environments where cloud provider node-naming conventions require it.

## Rationale

The --hostname-override flag changes the node name the kubelet registers with the API server. If set to another node's name, the kubelet could assume that node's identity and access its Secrets, ConfigMaps, and pod specifications, bypassing Node authorization. Cloud environments sometimes require the flag when the default OS hostname does not match the cloud provider's expected node identifier. Without governance, the flag can be set silently during node provisioning, creating an identity confusion vulnerability that is difficult to detect through normal monitoring.

## Scope

Applies to all production Kubernetes clusters. Covers the kubelet --hostname-override flag in kubelet configuration, systemd unit files, and node provisioning templates. For managed Kubernetes services, verify through provider documentation whether hostname override is used internally and how Node authorization identity binding is maintained.

## Enforcement

- Primary Enforcement Point: Node provisioning pipeline, kubelet configuration management, and periodic configuration audit scans.
- Implementation Surface: Kubelet provisioning templates, kubelet configuration management, exception registration process, and CIS Benchmark scanning (kube-bench check 4.2.8).
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Cloud Platform Engineering
