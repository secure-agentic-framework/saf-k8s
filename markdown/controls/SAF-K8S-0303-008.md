# SAF-K8S-0303-008 - AppArmor workload profile assignment and unconfined-mode restriction

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.3 - Mandatory Access Controls
- Maturity: Practitioner
- Classification: baseline

## Statement

Clusters using AppArmor-enabled nodes must assign the appropriate AppArmor profile to each protected workload and must not permit containers to run in unconfined mode unless a documented exception with compensating controls exists and is actively enforced.

## Objective

Enforce workload-specific AppArmor confinement by ensuring pods are explicitly assigned approved profiles and that unconfined execution remains prohibited except through controlled exceptions.

## Rationale

Loading profiles onto nodes is not sufficient if workloads are left unannotated or permitted to run unconfined. Workload-level assignment and exception control ensure AppArmor actually constrains file access, network operations, and capability use for the containers that need confinement.

## Scope

Applies to clusters running on AppArmor-capable nodes such as Ubuntu, Debian, SUSE, and other distributions that use AppArmor as the primary LSM. Does not apply to nodes using SELinux as the primary LSM.

## Enforcement

- Primary Enforcement Point: Pod specification validation, admission policy enforcement for AppArmor assignment, and exception governance for unconfined workloads.
- Implementation Surface: Pod annotations (container.apparmor.security.beta.kubernetes.io) or securityContext.appArmorProfile for profile assignment, policy engine rules validating required AppArmor profiles at admission, mutation or defaulting rules for runtime/default where appropriate, and documented exception workflows with compensating controls and periodic review for unconfined containers.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
