# SAF-K8S-0201-009 - Kubelet systemd unit hardening

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.1 - Kubelet and Node Configuration Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

The kubelet systemd unit file must be configured with security directives that limit the kubelet process's capabilities and filesystem access, including hardened systemd protections such as ProtectSystem, ProtectHome, NoNewPrivileges, PrivateTmp, and appropriate address-family restrictions, to reduce the blast radius of kubelet compromise.

## Objective

Prevent privilege escalation and limit the blast radius of kubelet process compromise by constraining the kubelet systemd unit to the minimum capabilities and filesystem access it requires.

## Rationale

The kubelet runs with broad host access by default. A compromised kubelet process or a kubelet restarted with attacker-controlled parameters can pivot to host resources outside Kubernetes control. Systemd security directives create OS-level enforcement boundaries that persist even when Kubernetes-level controls are bypassed. Audit verification is a separate detection obligation addressed by SAF-K8S-0201-010.

## Scope

Applies to all nodes running kubelet as a systemd-managed service. Managed Kubernetes services may satisfy this through provider hardening attestation and documentation of equivalent protections. Excludes node-level audit verification, which is addressed separately by SAF-K8S-0201-010.

## Enforcement

- Primary Enforcement Point: Node Service Hardening Through systemd Unit Configuration
- Implementation Surface: systemd unit file hardening directives, configuration management deployment of kubelet service definitions, and recurring validation of systemd service settings.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; OS/Infrastructure
