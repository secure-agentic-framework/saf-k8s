# SAF-K8S-0302-001 - Pod and container security context enforcement

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.2 - Security Contexts and Capabilities
- Maturity: Foundational
- Classification: baseline

## Statement

All containers must define security contexts that set runAsNonRoot to true, specify an explicit runAsUser with a non-zero UID, set readOnlyRootFilesystem to true, and set allowPrivilegeEscalation to false, unless a documented exception with compensating controls is approved.

## Objective

Constrain container runtime behavior to the minimum privilege required by enforcing non-root execution, immutable root filesystems, and privilege escalation prevention at the security context level.

## Rationale

Containers running as root can modify system binaries, access sensitive host paths, and exploit kernel vulnerabilities for container escape. Writable root filesystems allow attackers to persist malware and modify application binaries. Without allowPrivilegeEscalation set to false, setuid binaries or kernel exploits can grant full root capabilities even to containers started with reduced privileges.

## Scope

Applies to all pods and containers in production clusters, including init containers and sidecar containers. Managed services and third-party operator pods must comply or carry documented exceptions. System-level pods (kube-system) may require exceptions for specific fields.

## Enforcement

- Primary Enforcement Point: Pod Security Admission (Restricted level), supplemented by policy engine rules validating individual security context fields at admission time.
- Implementation Surface: Pod and container securityContext fields (runAsNonRoot, runAsUser, runAsGroup, readOnlyRootFilesystem, allowPrivilegeEscalation), Pod Security Admission (Restricted level), policy engine mutation and validation rules (OPA Gatekeeper, Kyverno) for injecting and enforcing security context defaults, emptyDir volume mounts for writable paths (/tmp, cache directories), and CI/CD manifest scanning for security context completeness.
- Owner Primary: Application Development teams
- Owner Supporting: Platform Engineering; Security Engineering
