# SAF-K8S-1005-018 - Component integrity verification before restoration after compromise

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.5 - Incident Response for Kubernetes
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must verify the integrity of components affected by an AI security incident before restoration to prevent reinfection or persistence during recovery operations.

## Objective

Ensure only verified components are restored after compromise so recovery does not reintroduce attacker persistence into the environment.

## Rationale

Restoring nodes, images, drivers, or other components without integrity verification can reintroduce the original compromise or preserve attacker footholds. Recovery must confirm component integrity before restored assets are trusted again.

## Scope

Applies to all Kubernetes clusters recovering from security incidents affecting AI workloads. Covers integrity verification of images, nodes, drivers, and related components before restoration to service.

## Enforcement

- Primary Enforcement Point: Post-incident restoration checklists and integrity verification workflows for components affected by the incident.
- Implementation Surface: Container image re-pull from trusted registries, hash or signature verification against known-good values, node OS integrity verification, GPU driver integrity checks, and other pre-restoration validation steps for affected components.
- Owner Primary: Security Operations
- Owner Supporting: Platform Engineering; AI/ML Engineering
