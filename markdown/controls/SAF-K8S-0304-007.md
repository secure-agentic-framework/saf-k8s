# SAF-K8S-0304-007 - Service account token automount opt-out

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.4 - Secure Defaults and Resource Constraints
- Maturity: Foundational
- Classification: baseline

## Statement

Pods that do not require Kubernetes API access must set automountServiceAccountToken to false at the ServiceAccount or pod level, preventing unnecessary API credentials from being mounted into containers.

## Objective

Reduce the attack surface of compromised containers by removing Kubernetes API credentials from pods that never use them, eliminating free credentials for API reconnaissance and privilege escalation.

## Rationale

Most application pods never call the Kubernetes API, yet by default Kubernetes mounts a service account token in every pod. After container compromise, an attacker can use this token for API reconnaissance, listing secrets, or escalating privileges through RBAC misconfigurations. Removing the token from pods that do not need it eliminates this credential exposure without affecting workload functionality.

## Scope

Applies to all workloads in production clusters that do not make Kubernetes API calls. Workloads that require API access (operators, controllers, monitoring agents) retain token mounts with least-privilege RBAC bindings.

## Enforcement

- Primary Enforcement Point: ServiceAccount configuration with automountServiceAccountToken set to false at the ServiceAccount level, overridden to true only for pods that require API access; policy engine rules enforcing opt-out by default.
- Implementation Surface: ServiceAccount and pod-level automountServiceAccountToken field set to false, policy engine rules enforcing automountServiceAccountToken false by default with exemptions for API-accessing workloads, and cluster scanning tools identifying pods with unnecessary service account tokens mounted.
- Owner Primary: Application Development teams
- Owner Supporting: Platform Engineering; Security Engineering
