# SAF-K8S-1001-004 - Audit volume management for AI workloads

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.1 - Logging and Audit
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Organizations must manage audit event volume for high-throughput AI workloads through selective audit policy design (tiered levels and omitStages to reduce low-value events at the source), audit backend tuning (batching, throttling, buffer sizing, and capacity planning), and continuous monitoring for event drops or backpressure, ensuring that security-critical AI events remain captured at full fidelity while the audit subsystem operates within capacity.

## Objective

Maintain audit system availability and completeness under the elevated API call volumes generated by AI training jobs and high-frequency inference workloads.

## Rationale

AI workloads, particularly distributed training jobs and high-throughput inference services, can generate API call volumes that overwhelm the Kubernetes audit subsystem, causing dropped audit records, increased API server latency, or audit backend failures. The Kubernetes audit policy model controls volume through level selection (None, Metadata, Request, RequestResponse) and omitStages, not through rate-limiting rules. Backend capacity is managed through webhook batching parameters (--audit-webhook-batch-max-size, --audit-webhook-batch-max-wait), log file rotation, and backend sizing. Without deliberate policy design and backend tuning, organizations face degraded cluster performance or audit gaps that conceal malicious activity.

## Scope

Applies to Kubernetes clusters running AI workloads that generate sustained high-volume API traffic, including distributed training orchestrators, batch inference services, and ML pipeline controllers. Audit policy and backend configuration must be reviewed when onboarding new high-throughput workloads.

## Enforcement

- Primary Enforcement Point: API server audit policy level and stage configuration, audit webhook batching and throttle parameters, audit backend capacity planning, and audit volume monitoring alerts.
- Implementation Surface: Kube-apiserver audit policy level and stage configuration (tiered None/Metadata/Request/RequestResponse levels, omitStages directives), user and service account audit exclusions (system:serviceaccount:kube-system:*), audit webhook batching parameters (--audit-webhook-batch-max-size, --audit-webhook-batch-max-wait, --audit-webhook-batch-buffer-size), audit log rotation flags (--audit-log-maxsize, --audit-log-maxbackup, --audit-log-maxage), audit backend capacity planning and sizing, separate audit sinks via log forwarders (Fluent Bit, Vector) for routing high-volume events to dedicated backends, and Prometheus monitoring of apiserver_audit_event_total, apiserver_audit_error_total, and apiserver_audit_requests_rejected_total metrics with alerting on dropped events and backpressure conditions.
- Owner Primary: Security Operations
- Owner Supporting: Platform Engineering; AI/ML Engineering
