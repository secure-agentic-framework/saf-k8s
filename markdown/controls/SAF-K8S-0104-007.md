# SAF-K8S-0104-007 - Emergency Kubernetes patch deployment procedures

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.4 - CIS Benchmarks and Patch Management
- Maturity: Foundational
- Classification: baseline

## Statement

Organizations must maintain documented emergency upgrade procedures for Kubernetes components (API server, kubelet, container runtime, and critical add-ons) that enable critical patches to be applied within organization-defined SLA windows, including pre-tested rollback plans, change communication processes, and post-patch validation steps.

## Objective

Minimize exploitation of known Kubernetes vulnerabilities by ensuring that emergency patches can be deployed rapidly, safely, and with operational continuity under time-pressure conditions.

## Rationale

When a critical CVE is identified, exploit code often follows within days. Without pre-established emergency upgrade procedures, patch deployment under time pressure introduces operational risk and delays. Documented runbooks, tested rollback procedures, and defined SLA windows enable rapid response without improvisation.

## Scope

Applies to all production and non-production clusters. Covers Kubernetes core component patches, container runtime patches, and critical add-on patches (CNI, CSI, ingress controllers). Also covers lifecycle management of cluster add-ons with independent release cycles (CoreDNS, CNI plugins, CSI drivers, ingress controllers, cert-manager).

## Enforcement

- Primary Enforcement Point: Governance / Process
- Implementation Surface: Emergency change management procedures, upgrade runbooks, rollback documentation, and post-patch validation scripts.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; Change Management; Incident Response
