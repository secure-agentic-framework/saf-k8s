# SAF-K8S-0904-013 - Pipeline artifact storage encryption and object-store access policy enforcement

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.4 - AI Pipeline Orchestration and Experimentation Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Pipeline artifacts, including intermediate data, model artifacts, and pipeline metadata, must be stored with encryption at rest and explicit object-storage access policies that restrict artifact retrieval and modification to approved identities and pipeline contexts.

## Objective

Protect pipeline artifact stores from unauthorized disclosure or tampering through storage-layer encryption and object-store policy enforcement.

## Rationale

ML pipelines generate artifacts that frequently contain sensitive data subsets, features, model weights, and execution metadata. Without storage encryption and explicit access policies on the artifact store, any pod or identity with broad storage reach can read or alter those artifacts outside the intended pipeline flow.

## Scope

Applies to object storage and persistent artifact repositories used by ML pipelines on Kubernetes, including MinIO, S3, GCS, and equivalent services. Covers encryption at rest, bucket or object policy enforcement, and artifact-store access restriction. Excludes stage isolation and cross-classification authorization, which are addressed separately.

## Enforcement

- Primary Enforcement Point: Object storage bucket policy, KMS configuration, and artifact store access-control settings.
- Implementation Surface: Encrypted object storage, per-bucket or per-prefix access policies, artifact repository IAM bindings, versioning where used for artifact traceability, and storage-side controls restricting read or write access to approved pipeline identities.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Data Engineering; Security Engineering
