# SAF-K8S-0301-002 - Pod Security Admission configuration and version pinning

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.1 - Pod Security Standards and Admission
- Maturity: Foundational
- Classification: baseline

## Statement

Production clusters must configure Pod Security Admission with namespace labels specifying enforcement mode (enforce), audit mode, and warn mode at explicit Kubernetes version pins, and must not rely on unpinned or floating version defaults.

## Objective

Guarantee predictable and auditable pod admission behavior by explicitly configuring PSA enforcement modes and pinning them to known Kubernetes API versions, preventing unexpected policy changes during cluster upgrades.

## Rationale

PSA enforcement modes without explicit version pinning may silently change behavior when the cluster is upgraded, potentially admitting previously blocked workloads or rejecting legitimate ones. Missing audit and warn modes eliminate visibility into policy violations before enforcement, increasing the risk of service disruptions during policy rollouts.

## Scope

Applies to all namespaces in production and staging clusters. Version pinning is required for enforce and audit modes. Managed Kubernetes services must have PSA labels verified after provider-initiated upgrades.

## Enforcement

- Primary Enforcement Point: Namespace metadata labels (pod-security.kubernetes.io/enforce, pod-security.kubernetes.io/enforce-version, audit, warn variants) validated by admission control and GitOps pipeline checks.
- Implementation Surface: Namespace metadata labels (pod-security.kubernetes.io/enforce-version, audit-version, warn-version), Pod Security Admission controller version-pin configuration, cluster upgrade checklists and automation scripts, GitOps pipeline validation for PSA label presence and version correctness, and PSS profile changelog review tooling.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Release Engineering
