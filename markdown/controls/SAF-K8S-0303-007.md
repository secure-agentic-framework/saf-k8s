# SAF-K8S-0303-007 - AppArmor profile distribution and node readiness

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.3 - Mandatory Access Controls
- Maturity: Practitioner
- Classification: baseline

## Statement

Clusters using AppArmor-enabled nodes must distribute approved AppArmor profiles to all applicable nodes through provisioning automation and must verify that each node pool consistently loads the required profiles before workloads that depend on them are scheduled.

## Objective

Ensure AppArmor protections are actually available on every applicable node by making profile distribution and node readiness a managed platform control rather than a best-effort host configuration task.

## Rationale

AppArmor enforcement fails silently when required profiles are missing from some nodes. Consistent profile distribution and node readiness verification prevent scheduling gaps where a workload lands on a node that cannot enforce its expected AppArmor confinement.

## Scope

Applies to clusters running on AppArmor-capable nodes such as Ubuntu, Debian, SUSE, and other distributions that use AppArmor as the primary LSM. Does not apply to nodes using SELinux as the primary LSM. Profile loading must be consistent across all nodes in a node pool.

## Enforcement

- Primary Enforcement Point: Node provisioning automation, machine image preparation, DaemonSet-based profile loading, and node readiness verification for AppArmor profile availability.
- Implementation Surface: AppArmor profile files loaded on nodes via DaemonSets, cloud-init, Ansible, or machine image baking; node provisioning automation for consistent profile distribution; node-level AppArmor verification such as sys/module/apparmor/parameters/enabled and aa-status; and scheduling gates or readiness checks that confirm required profiles are present before dependent workloads run.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development teams
