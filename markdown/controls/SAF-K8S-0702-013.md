# SAF-K8S-0702-013 - Tenant default-deny inter-namespace network isolation

- Domain: D07 - Storage, Multi-tenancy, and Resource Governance
- Knowledge Area: 7.2 - Namespace Isolation and Multi-Tenancy
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must enforce tenant traffic isolation with default-deny ingress and egress NetworkPolicy in every tenant namespace, permitting inter-namespace communication only through explicitly documented and reviewed allow rules.

## Objective

Prevent lateral movement and cross-tenant runtime access by making all traffic paths between tenant namespaces and shared services explicit and auditable.

## Rationale

Namespace scoping and RBAC do not constrain pod-to-pod traffic once workloads are running. Without default-deny network policy, a compromised tenant workload can laterally move to peer namespaces or access shared services even when API permissions are correctly scoped.

## Scope

Applies to all multi-tenant Kubernetes clusters. Every tenant namespace must implement default-deny ingress and egress controls. Exceptions for shared services, monitoring, ingress, or pipeline dependencies must be explicitly allowlisted, documented, and periodically reviewed.

## Enforcement

- Primary Enforcement Point: Kubernetes NetworkPolicy objects and network isolation validation tests for inter-namespace communication paths.
- Implementation Surface: Default-deny ingress and egress NetworkPolicy objects per tenant namespace, explicit allow rules for approved shared services and workflows, exception documentation, continuous connectivity testing for denied and approved paths, and periodic penetration testing of tenant isolation boundaries.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Network Engineering
