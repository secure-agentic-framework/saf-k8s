# SAF-K8S-0505-005 - Identity-based internal service access control

- Domain: D05 - Network Security and Communication
- Knowledge Area: 5.5 - API Server and Service Exposure Protection
- Maturity: Practitioner
- Classification: baseline

## Statement

Access to internal service endpoints must be governed by service account-based authentication and authorization patterns rather than relying solely on network reachability, using service mesh authorization policies or application-level authentication for sensitive internal services.

## Objective

Ensure that network reachability alone does not constitute authorization for internal service access, requiring identity-based verification for sensitive services in multi-tenant or zero trust environments.

## Rationale

Relying solely on network reachability for internal service authorization means any pod in an allowed network segment can access any service, which is insufficient in multi-tenant or zero trust environments. Identity-based access control binds authorization to workload identity (service accounts) rather than network location, enabling fine-grained access decisions for sensitive internal services such as model registries, experiment trackers, feature stores, and GPU metrics exporters.

## Scope

Applies to production clusters where service mesh or application-level authentication is available. Priority should be given to sensitive internal services including model registries, experiment trackers, feature stores, distributed training rendezvous endpoints, and GPU metrics exporters.

## Enforcement

- Primary Enforcement Point: Service mesh authorization policies for service account-based access control (Istio AuthorizationPolicy, Linkerd ServerAuthorization) or application-level authentication and authorization.
- Implementation Surface: Service mesh authorization policies for service account-based access control (Istio AuthorizationPolicy with source principal matching, Linkerd ServerAuthorization with ServiceAccount selectors), application-level authentication and authorization for sensitive internal services (model registry, experiment tracker, feature store, distributed training rendezvous, GPU metrics exporter), and service account token projection for workload identity attestation.
- Owner Primary: Security Engineering
- Owner Supporting: Platform Engineering; Application Development
