# SAF-K8S-1002-001 - Metric endpoint authentication

- Domain: D10 - Observability, Incident Response, and Governance
- Knowledge Area: 10.2 - Monitoring, Metrics, and Tracing
- Maturity: Practitioner
- Classification: baseline

## Statement

Organizations must enforce authentication on Prometheus, metrics-server, and other metric endpoints to prevent unauthenticated access to cluster and AI workload operational metrics.

## Objective

Prevent unauthenticated scraping or query access to metric endpoints that expose cluster topology, resource utilization, and AI workload performance data.

## Rationale

Unauthenticated metric endpoints expose cluster topology, resource utilization patterns, and AI workload performance data that adversaries can use for reconnaissance or to calibrate denial-of-service attacks against GPU resources.

## Scope

Applies to all metric collection and exposition endpoints in Kubernetes clusters hosting AI workloads, including Prometheus, metrics-server, and custom metric endpoints. Covers both cluster-level and namespace-level metric access controls.

## Enforcement

- Primary Enforcement Point: Metric endpoint authentication configuration and RBAC policies for metric access.
- Implementation Surface: Prometheus server authentication configuration (--web.config.file for basic auth or TLS client certificates), managed Prometheus IAM-based authentication (Amazon Managed Prometheus, Google Cloud Managed Service for Prometheus), metrics-server RBAC (system:metrics-reader ClusterRole binding restrictions), custom metrics API RBAC (custom.metrics.k8s.io, external.metrics.k8s.io), kubelet /metrics and /metrics/cadvisor endpoint authorization (--authorization-mode=Webhook), NetworkPolicy restrictions limiting scrape target access to Prometheus pods, and Prometheus recording rules and relabeling for sensitive metric redaction (GPU topology, training hyperparameters).
- Owner Primary: Security Operations
- Owner Supporting: Platform Engineering; AI/ML Engineering
