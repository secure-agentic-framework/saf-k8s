# SAF-K8S-0604-008 - AI workload vulnerability exposure classification

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.4 - SBOMs and Vulnerability Intelligence
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Vulnerability management processes must distinguish training-time from inference-time exposure and separately classify GPU driver and CUDA stack CVEs from application-layer CVEs based on AI workload context before remediation decisions are made.

## Objective

Ensure AI workload vulnerabilities are evaluated in their actual exploitation context rather than as an undifferentiated queue of findings.

## Rationale

Generic severity scoring does not capture the operational context of AI workloads. A GPU driver flaw in a training-only environment, an ML framework library flaw in an internet-facing inference endpoint, and a preprocessing dependency issue in an internal pipeline produce different attack surfaces and different urgency. Without explicit exposure classification, downstream remediation decisions are noisy and inconsistent.

## Scope

Applies to all AI and ML workload images and runtime environments deployed in Kubernetes, including inference servers, training jobs, data preprocessing pipelines, and model-serving infrastructure. Covers application-layer dependencies and GPU/CUDA driver stack vulnerabilities. Does not define remediation SLAs or automated rebuild execution, which are addressed by SAF-K8S-0604-009 and SAF-K8S-0604-010.

## Enforcement

- Primary Enforcement Point: Vulnerability management platform classification policy configuration and reporting views that segment findings by AI workload context.
- Implementation Surface: Vulnerability management platform policy configuration distinguishing training-time versus inference-time exposure, GPU driver and CUDA stack CVEs versus application-layer CVEs, cluster- and workload-tier metadata enrichment, and reporting or dashboard views that segment AI workload findings by exposure class before remediation routing.
- Owner Primary: Security Engineering
- Owner Supporting: ML Platform Engineering; DevSecOps / Release Engineering
