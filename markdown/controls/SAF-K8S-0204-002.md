# SAF-K8S-0204-002 - Kubernetes-specific runtime detection rules

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.4 - Runtime Threat Detection
- Maturity: Practitioner
- Classification: baseline

## Statement

Runtime security tools must include detection rules specifically tuned for Kubernetes-native threats including container escape attempts, in-container privilege escalation, lateral movement between pods or nodes, and cryptomining indicators such as unexpected CPU-intensive processes or connections to known mining pool endpoints.

## Objective

Ensure that runtime detection capabilities address the specific threat landscape of containerized Kubernetes environments rather than relying solely on generic host-based detection signatures.

## Rationale

Generic host-based detection rules miss Kubernetes-specific attack patterns such as nsenter-based container escape, /proc/self/exe exploitation, service account token theft for lateral movement, and cryptomining processes that disguise themselves within container namespaces. Without purpose-built detection rules, Kubernetes-native attacks can progress through kill chain stages undetected. Regular rule tuning is necessary as new container escape CVEs and Kubernetes-specific exploitation techniques emerge.

## Scope

Applies to all runtime security tool deployments on production cluster nodes. Detection rules must be reviewed and updated at least quarterly or upon publication of new Kubernetes-relevant CVEs. Rules must be tested in non-production environments before deployment to avoid false positive disruption in production workloads.

## Enforcement

- Primary Enforcement Point: Runtime security tool rule configuration, managed through version-controlled rule repositories and automated deployment pipelines.
- Implementation Surface: Runtime detection rule definitions (Falco rules, Tetragon TracingPolicy), detection categories (container escape, privilege escalation, lateral movement, cryptomining, credential theft), MITRE ATT&CK mapping, rule testing frameworks, and quarterly rule review procedures.
- Owner Primary: Security Engineering
- Owner Supporting: Security Operations (SOC); Threat Intelligence
