# SAF-K8S-0102-021 - etcd certificate expiration monitoring and lead-time alerting

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Practitioner
- Classification: baseline

## Statement

Production Kubernetes clusters must generate monitoring alerts with sufficient lead time to remediate etcd server, peer, and API server-to-etcd client certificates approaching expiration.

## Objective

Prevent etcd certificate outages by ensuring expiration telemetry and alert routing provide actionable remediation lead time.

## Rationale

Even with appropriate validity limits, etcd availability is still at risk if approaching certificate expiration is not visible to operators early enough to renew or replace certificates safely. Dedicated monitoring and alerting make certificate expiry operationally actionable before it becomes a control-plane outage.

## Scope

Applies to all self-managed Kubernetes clusters using etcd. Covers expiration telemetry, dashboards, threshold policy, and notification routing for etcd-related certificates.

## Enforcement

- Primary Enforcement Point: Certificate expiration monitoring and alert routing
- Implementation Surface: Expiration dashboards, alert rules, notification routing, and periodic alert-path validation for etcd certificate expiry.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering
