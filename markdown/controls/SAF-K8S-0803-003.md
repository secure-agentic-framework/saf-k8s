# SAF-K8S-0803-003 - RDMA memory region and queue pair isolation

- Domain: D08 - GPU, Accelerator, and Confidential Computing
- Knowledge Area: 8.3 - High-Performance Interconnect Security
- Maturity: Advanced
- Classification: ai-specific

## Statement

RDMA-enabled AI workloads must enforce memory region access controls with properly scoped remote access keys, isolate queue pairs between tenants, and implement protections against remote memory read/write attacks targeting GPU training buffers.

## Objective

Prevent unauthorized remote direct memory access to GPU training data, model parameters, and gradient buffers by enforcing strict RDMA memory region boundaries and queue pair isolation between workloads.

## Rationale

RDMA allows remote nodes to read and write memory without CPU involvement, using memory registration keys (rkeys) for access control. A compromised or malicious workload that obtains a valid rkey can silently read training data or write corrupted gradients to a victim's memory regions. Shared queue pairs between tenants can enable traffic injection into another tenant's RDMA communication streams, poisoning distributed training collective operations.

## Scope

Applies to Kubernetes pods using RDMA (InfiniBand verbs, RoCE) for distributed training communication libraries (NCCL, Gloo, MPI). Covers memory region registration, protection domain configuration, and queue pair allocation. Workloads not using RDMA are not in scope. Managed RDMA services must verify provider-enforced memory isolation.

## Enforcement

- Primary Enforcement Point: RDMA device plugin configuration, libibverbs memory region registration policies, protection domain isolation in RDMA workloads, and kernel-level RDMA access controls (rdma cgroup).
- Implementation Surface: RDMA verbs layer memory region (MR) registration and remote key (rkey) scoping, RDMA queue pair (QP) creation and isolation per training job, RDMA protection domain assignment per tenant, libibverbs memory region registration policies, rdma cgroup controller configuration restricting per-container RDMA resource allocation, NCCL communication channel isolation settings, and RDMA device plugin per-pod protection domain configuration.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Network Engineering; GPU Infrastructure Engineering; Security Engineering
