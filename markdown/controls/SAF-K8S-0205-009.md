# SAF-K8S-0205-009 - Service proxy path hardening for kube-proxy or replacements

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.5 - kube-proxy and Node Networking Security
- Maturity: Practitioner
- Classification: baseline

## Statement

Clusters must harden the selected service proxy path by restricting kube-proxy or replacement permissions, enabling strict conntrack or equivalent state controls, and removing unnecessary service proxy capabilities.

## Objective

Reduce attack surface in the active service proxy implementation after the mode or replacement choice is made.

## Rationale

Even after a mode is selected, permissive kube-proxy privileges, loose conntrack behavior, or unnecessary service proxy capabilities can widen attack surface or undermine network policy expectations. Hardening the chosen implementation is a distinct obligation from choosing the implementation in the first place.

## Scope

Applies to all production clusters using kube-proxy or kube-proxy replacement implementations. Clusters using eBPF-based CNI plugins that fully replace kube-proxy (e.g., Cilium kube-proxy replacement) must document the replacement and apply equivalent hardening. Managed Kubernetes services must document the provider-selected mode and any available hardening options.

## Enforcement

- Primary Enforcement Point: kube-proxy ConfigMap or DaemonSet configuration, replacement DaemonSet configuration, and node networking hardening settings for the active service proxy path.
- Implementation Surface: kube-proxy RBAC, conntrack tuning, kube-proxy ConfigMap or replacement DaemonSet settings, and removal of unnecessary proxy capabilities in the selected service routing path.
- Owner Primary: Platform Engineering
- Owner Supporting: Network Engineering; Security Engineering
