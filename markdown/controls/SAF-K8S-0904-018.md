# SAF-K8S-0904-018 - Pipeline definition immutable version storage and controlled rollback governance

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.4 - AI Pipeline Orchestration and Experimentation Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Pipeline definitions must be stored as immutable versions and changed only through controlled rollback procedures that preserve version integrity and prevent silent reintroduction of tampered ML workflows.

## Objective

Preserve recoverable, tamper-resistant pipeline definition history so trusted rollback is possible without undermining workflow integrity.

## Rationale

Signing alone does not guarantee safe recovery if prior workflow definitions can be overwritten or rollback is performed without governance. Immutable version history and controlled rollback procedures ensure teams can recover to known-good pipeline definitions without reintroducing unreviewed or silently modified workflows.

## Scope

Applies to ML pipeline definitions, templates, and workflow manifests stored and executed on Kubernetes-based orchestration platforms such as Argo Workflows and Kubeflow Pipelines. Covers immutable version storage, retained version history, and governed rollback procedures. Excludes cryptographic signing and verification, which are addressed separately.

## Enforcement

- Primary Enforcement Point: Pipeline definition version repository and rollback process.
- Implementation Surface: Immutable workflow or template version storage, protected version history, rollback authorization or approval records, and recovery evidence showing rollback uses known-good preserved definitions rather than mutable replacements.
- Owner Primary: ML Platform Engineering
- Owner Supporting: Security Engineering; ML Engineering
