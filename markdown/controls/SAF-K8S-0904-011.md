# SAF-K8S-0904-011 - Pipeline stage isolation between sensitivity levels

- Domain: D09 - AI Workload Security: Training, Serving, and Pipelines
- Knowledge Area: 9.4 - AI Pipeline Orchestration and Experimentation Security
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Pipeline steps must be isolated to prevent data leakage between stages operating at different sensitivity levels.

## Objective

Prevent unauthorized data exposure between pipeline stages by enforcing execution and communication isolation across differing sensitivity levels.

## Rationale

Without effective stage isolation, a lower-sensitivity pipeline step can read artifacts, intermediate data, or side effects from a higher-sensitivity stage, undermining data-separation guarantees and enabling cross-stage leakage.

## Scope

Applies to ML pipeline orchestration on Kubernetes where different steps process data of different sensitivity levels. Covers namespace or runtime isolation between stages, restricted cross-stage communication paths, and leakage testing between isolated pipeline steps. Excludes explicit authorization for cross-classification transfers, which is covered by SAF-K8S-0904-012.

## Enforcement

- Primary Enforcement Point: Pipeline orchestrator isolation settings and namespace or network boundary controls.
- Implementation Surface: Namespace isolation between stages, per-stage runtime separation, restricted service-account and network paths between pipeline steps, and leakage testing that verifies lower-sensitivity stages cannot access higher-sensitivity stage data or artifacts.
- Owner Primary: Data Engineering
- Owner Supporting: ML Platform Engineering; Security Engineering
