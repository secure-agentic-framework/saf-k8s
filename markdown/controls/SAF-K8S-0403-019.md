# SAF-K8S-0403-019 - Kubernetes Secrets external KMS provider integration

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.3 - Secrets Management
- Maturity: Foundational
- Classification: baseline

## Statement

Kubernetes Secrets encryption at rest must use an external KMS provider as the key management backend.

## Objective

Ensure Kubernetes Secrets encryption relies on a key management service outside the cluster trust boundary.

## Rationale

Externalizing key management is a distinct control claim from governing the permissions on the key itself. This control isolates KMS backend integration so teams can independently verify that Kubernetes Secrets encryption uses an external KMS provider or customer-managed key path.

## Scope

Applies to all production Kubernetes clusters. Covers KMS provider selection and integration, Kubernetes EncryptionConfiguration KMS backends, KMS v2 adoption, and customer-managed key configuration for managed Kubernetes services. Key policy restrictions remain governed by SAF-K8S-0403-020, and the underlying API-resource encryption mechanism remains governed by SAF-K8S-0101-003.

## Enforcement

- Primary Enforcement Point: Kubernetes EncryptionConfiguration and managed-service customer-managed key integration settings.
- Implementation Surface: Kubernetes EncryptionConfiguration with KMS provider plugins, KMS v2 API integration, and customer-managed key configuration for managed Kubernetes services.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Key Management/Cryptography; Cloud Platform Engineering
