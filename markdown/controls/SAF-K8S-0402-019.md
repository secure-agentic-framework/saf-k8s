# SAF-K8S-0402-019 - Dedicated workload service accounts and least-privilege assignment

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.2 - Service Accounts and Workload Identity
- Maturity: Foundational
- Classification: baseline

## Statement

Each workload must be assigned a dedicated service account that is scoped to the workload's function and bound only to the permissions required for that workload.

## Objective

Ensure each workload operates with a distinct Kubernetes identity and least-privilege authorization instead of inheriting a shared namespace credential.

## Rationale

Per-workload service accounts create clear identity boundaries between applications and limit the blast radius of token compromise. Dedicated identities also support auditable authorization review and workload-specific permission minimization.

## Scope

Applies to all application workloads in production and pre-production clusters, including training, inference, and batch jobs that require Kubernetes API or cloud identity access.

## Enforcement

- Primary Enforcement Point: Workload specification and provisioning controls that require explicit serviceAccountName assignment and scoped RBAC or workload identity bindings for each workload class.
- Implementation Surface: Pod specs, service account provisioning automation, RBAC bindings, workload identity bindings, and deployment policy checks enforcing explicit dedicated service accounts.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; Application Development
