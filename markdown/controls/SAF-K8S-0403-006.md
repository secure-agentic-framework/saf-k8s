# SAF-K8S-0403-006 - Per-workload credential scoping for AI jobs

- Domain: D04 - Identity, Access, and Secrets Management
- Knowledge Area: 4.3 - Secrets Management
- Maturity: Practitioner
- Classification: ai-specific

## Statement

Each AI workload (training job, inference server, pipeline step) must receive credentials scoped to exactly the resources it requires -- specific storage prefixes, model registry repositories, feature store tables, and API endpoints -- and must not share broadly-scoped platform credentials across workloads with different data access requirements.

## Objective

Prevent lateral movement and blast radius expansion by ensuring that a compromised AI workload's credentials grant access only to the specific resources that workload legitimately needs, not to the entire AI platform's data and services.

## Rationale

AI platforms commonly use a single "ML platform" service account or IAM role with broad access to data lakes, model registries, feature stores, and compute resources. When thousands of training jobs share one credential scope, any compromised job gains access to all training data, all models, and all downstream services. Per-workload scoping limits blast radius to the specific datasets, model repositories, and APIs that each job actually needs. SAF-K8S-0402-008 establishes service account segmentation by workload type; this control extends that principle to the credential scope granted to each individual job.

## Scope

Applies to all AI/ML workloads including training jobs, inference servers, pipeline orchestrator steps, notebook sessions, and batch evaluation jobs. Covers cloud IAM policies, Vault dynamic secret policies, Kubernetes RBAC bindings, and external service API key scoping. Complements SAF-K8S-0402-008 (workload type segmentation) and SAF-K8S-0402-009 (ephemeral credential lifecycle).

## Enforcement

- Primary Enforcement Point: Cloud IAM policy conditions, Vault policy paths, Kubernetes RBAC bindings, and admission policies validating credential scope annotations.
- Implementation Surface: Cloud IAM (AWS IAM policy conditions, GCP IAM conditions, Azure RBAC conditions), HashiCorp Vault policies with templated paths, Kubernetes service accounts with per-job RBAC bindings, pipeline orchestrators (Kubeflow, Argo, Ray) job templates.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; ML Engineering; Cloud Platform Engineering
