# SAF-K8S-0605-008 - Admission webhook TLS rotation and high-availability resilience

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.5 - Admission Control
- Maturity: Practitioner
- Classification: baseline

## Statement

Admission webhooks must use automatically rotated TLS certificates and must be deployed with high-availability and monitoring configurations that prevent certificate expiration, backend outages, or sustained latency errors from making the webhook unavailable to the cluster.

## Objective

Keep webhook trust chains valid and webhook backends continuously available so admission enforcement infrastructure does not become a single point of failure for cluster operations.

## Rationale

Expired webhook certificates, single-replica backends, and absent latency monitoring can block admission entirely or create prolonged control-plane outages. Automated certificate lifecycle management and resilient deployment topology reduce that operational fragility.

## Scope

Applies to all validating and mutating admission webhooks in production and staging Kubernetes clusters, including custom webhooks and framework-provided webhooks such as Gatekeeper, Kyverno, and Kubewarden.

## Enforcement

- Primary Enforcement Point: Webhook deployment infrastructure and certificate lifecycle management.
- Implementation Surface: Automated TLS certificate rotation, CA injection, webhook backend Deployment replicas, PodDisruptionBudgets, anti-affinity settings, and webhook latency and availability dashboards with alerting.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; SRE / Operations
