# SAF-K8S-0201-014 - Kubelet serving certificate trust and expiry enforcement

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.1 - Kubelet and Node Configuration Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Kubelet HTTPS endpoints must use valid serving certificates issued by a trusted cluster or organizational certificate authority, and expired or self-signed serving certificates must not remain in service.

## Objective

Maintain trustworthy kubelet HTTPS endpoints for node management traffic, exec, logs, and metrics access.

## Rationale

Expired or self-signed kubelet serving certificates enable man-in-the-middle attacks and break trust in kubelet HTTPS endpoints. Enforcing trusted issuance and expiry replacement preserves authenticated node-management traffic and prevents insecure fallback behavior.

## Scope

Applies to all cluster nodes running kubelet, including control-plane nodes where kubelet manages static pods. For managed services that handle certificate lifecycle internally, evidence of provider-managed rotation and TLS enforcement satisfies this control. Environments using external certificate authorities must document their rotation schedule and monitoring.

## Enforcement

- Primary Enforcement Point: Kubelet serving certificate bootstrap configuration, serving CSR approval workflow, and certificate chain or expiry validation.
- Implementation Surface: KubeletConfiguration serverTLSBootstrap setting, kubelet-serving CSR approver, Kubernetes certificates API, and serving certificate chain and expiry validation tooling.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; PKI/Certificate Management
