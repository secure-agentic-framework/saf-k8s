# SAF-K8S-0201-013 - Kubelet client certificate rotation via TLS bootstrap

- Domain: D02 - Node, Runtime, and OS Security
- Knowledge Area: 2.1 - Kubelet and Node Configuration Hardening
- Maturity: Foundational
- Classification: baseline

## Statement

Kubelet instances must use TLS bootstrapping and automatic client certificate rotation so node identities are issued by the cluster certificate authority and renewed before expiry without manual credential reuse.

## Objective

Maintain current kubelet client identity for node-to-API-server authentication.

## Rationale

Kubelet client certificates establish each node's authenticated identity to the API server. TLS bootstrap and automatic rotation prevent expired credentials, reduce manual renewal drift, and ensure node identities remain bound to approved certificate issuance workflows.

## Scope

Applies to all cluster nodes running kubelet, including control-plane nodes where kubelet manages static pods. For managed services that handle certificate lifecycle internally, evidence of provider-managed rotation and TLS enforcement satisfies this control. Environments using external certificate authorities must document their rotation schedule and monitoring.

## Enforcement

- Primary Enforcement Point: Kubelet bootstrap configuration, certificate signing workflow, and client certificate rotation monitoring.
- Implementation Surface: KubeletConfiguration rotateCertificates setting, bootstrap kubeconfig or bootstrap tokens, Kubernetes certificates API, and CSR approval workflow.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; PKI/Certificate Management
