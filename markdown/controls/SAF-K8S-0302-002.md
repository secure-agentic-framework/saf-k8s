# SAF-K8S-0302-002 - Linux capability drop-all and least-privilege add-back

- Domain: D03 - Workload and Pod Security
- Knowledge Area: 3.2 - Security Contexts and Capabilities
- Maturity: Foundational
- Classification: ai-specific

## Statement

All containers must drop ALL Linux capabilities and add back only the specific capabilities required for the workload to function, with GPU workloads documenting which elevated capabilities (e.g., SYS_ADMIN, SYS_RAWIO) are required for device access and whether least-privilege alternatives (device plugins, CDI) can replace them.

## Objective

Minimize the kernel-level privileges available to containers by enforcing a drop-all-then-add-back capability model, preventing unnecessary access to dangerous kernel interfaces including those commonly over-provisioned for GPU workloads.

## Rationale

Linux capabilities grant fine-grained kernel privileges that attackers can use for container escape, network manipulation, or host filesystem access. GPU workloads are frequently granted SYS_ADMIN or SYS_RAWIO for device access, but these capabilities also enable mount namespace manipulation and raw I/O to any device. Without a drop-all baseline, containers inherit a default capability set that includes NET_RAW, enabling network-level attacks within the cluster.

## Scope

Applies to all containers in production clusters. GPU and accelerator workloads must document required capabilities and demonstrate that device-plugin or CDI-based alternatives have been evaluated before elevated capabilities are approved.

## Enforcement

- Primary Enforcement Point: Pod Security Admission (Restricted level requires drop ALL), policy engine rules validating capabilities.add lists against an approved allowlist per workload class.
- Implementation Surface: Container securityContext.capabilities (drop: ALL, add: specific capabilities), Pod Security Admission (Restricted level), policy engine allowlists for per-workload-class capability grants, NVIDIA device plugins with CDI (Container Device Interface) support (--cdi-enabled flag), containerd 1.7+/CRI-O 1.28+ CDI runtime support, strace and seccomp log profiling for capability verification, GPU driver version documentation, and InfiniBand/RDMA capability scoping (IPC_LOCK, NET_RAW) for multi-GPU training pods.
- Owner Primary: Application Development teams
- Owner Supporting: Platform Engineering; AI/ML Platform team; Security Engineering
