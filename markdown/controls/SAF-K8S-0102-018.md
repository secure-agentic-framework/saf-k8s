# SAF-K8S-0102-018 - etcd client and peer certificate authentication

- Domain: D01 - Control Plane and Cluster Hardening
- Knowledge Area: 1.2 - etcd and Cluster State Protection
- Maturity: Foundational
- Classification: baseline

## Statement

Production Kubernetes clusters must enforce client certificate authentication for all etcd client connections and mutual TLS with certificate-based peer authentication for all etcd member communication.

## Objective

Prevent unauthorized etcd access and spoofed member communication by requiring certificate-based authentication on all etcd client and peer channels.

## Rationale

etcd holds the authoritative cluster state. Without client certificate authentication, any network-reachable process can attempt direct etcd access. Without mutually authenticated peer TLS, inter-member traffic can be intercepted or spoofed, undermining cluster-state integrity and availability.

## Scope

Applies to all production Kubernetes clusters. Covers API server to etcd client authentication and etcd member-to-member peer authentication. For managed Kubernetes services, applies to verifying that provider-managed etcd uses customer-inaccessible, certificate-authenticated channels or equivalent provider controls.

## Enforcement

- Primary Enforcement Point: etcd transport authentication
- Implementation Surface: etcd configuration flags (--client-cert-auth, --peer-client-cert-auth, --cert-file, --peer-cert-file), certificate issuance and trust configuration, and periodic certificate validation testing.
- Owner Primary: Platform Engineering
- Owner Supporting: Security Engineering; PKI / Certificate Operations
