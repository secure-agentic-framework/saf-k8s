# SAF-K8S-0604-013 - Hermetic build execution and pinned dependency input control

- Domain: D06 - Supply Chain, Images, and Admission Control
- Knowledge Area: 6.4 - SBOMs and Vulnerability Intelligence
- Maturity: Practitioner
- Classification: baseline

## Statement

Build pipelines must use hermetic build configurations that prevent uncontrolled network access or live dependency resolution during build execution and must enforce committed dependency lock files and pinned dependency inputs.

## Objective

Prevent dependency confusion, typosquatting, and uncontrolled build-time input changes by ensuring builds execute from approved, pre-resolved, and pinned dependency sources.

## Rationale

Non-hermetic builds that resolve dependencies dynamically can ingest malicious or unexpected content at build time even when provenance exists. Hermetic execution and pinned dependency inputs reduce that attack surface and make dependency state reproducible for review.

## Scope

Applies to all CI/CD pipelines producing deployable Kubernetes artifacts, including application images, AI workload images, and infrastructure operator images. Approved mirrors, proxies, or caches may be used when they preserve pinned dependency and integrity guarantees.

## Enforcement

- Primary Enforcement Point: CI/CD build environment configuration and dependency resolution controls.
- Implementation Surface: Hermetic build settings, no-egress build workers, approved dependency mirrors, committed lock files with hash validation, and enforcement of pinned dependency inputs during build execution.
- Owner Primary: DevSecOps / Release Engineering
- Owner Supporting: Security Engineering; Platform Engineering
